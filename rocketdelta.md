# Olive Link Collective

Olive Link Collective 是一个面向开发者、技术研究人员与开源项目维护者的结构化外链资源汇总系统。该项目不提供传统意义上的代码库或框架，而是以高可读性的 Markdown 文档为载体，对分散在互联网各处的优质技术文章、工具文档、架构复盘与工程实践进行人工筛选、逻辑归类和语义索引。项目定位为“技术决策者的外脑”，帮助用户在信息过载的环境中快速定位到高信噪比的原始资料，减少重复搜索与无效阅读的时间损耗。

目标用户包括但不限于：基础架构工程师、SRE 团队、技术负责人、开源贡献者以及计算机科学相关专业的高年级学生。项目通过建立标签体系、上下游依赖关系图谱与版本时间线标注，使每个外链资源不再是孤立页面，而成为可追溯、可对比、可复用的知识节点。

## 功能概览

- **结构化资源索引**：所有外链资源按技术领域、应用场景与成熟度等级进行多维度分类，每个条目附带摘要标签与阅读时长估计。

- **版本化快照引用**：针对 GitHub 仓库内的 Markdown 文档，项目会记录其提交哈希与最后更新时间，方便用户追溯内容变迁。

- **关联图谱浏览**：资源之间通过“前置阅读”“延伸材料”“替代方案”等关系相互连接，形成网状知识结构。

- **离线缓存镜像**：提供核心文档的 PDF 与 Markdown 静态导出，适用于内网环境或网络受限场景。

- **关键词全文检索**：基于简单的前端索引表，支持对资源标题、摘要、标签与正文首段进行快速匹配。

- **社区反馈集成**：每个资源条目可附带星级评分、阅读笔记与争议标记，帮助新用户快速判断内容可靠性。

- **自动化元数据抽取**：通过解析 URL 模式与页面标题规则，自动补全文档所属组织、作者信息与发布平台。

## 应用场景

- **技术选型前的调研准备**：当架构师需要对比多个中间件或存储方案时，可使用本项目的“方案对比”分类快速获取各产品的官方文档、性能测试报告与社区踩坑记录，大幅缩短调研周期。

- **新成员入职培训路径规划**：团队 Leader 可依据本项目的“入门序列”标签，为新员工整理一份从基础概念到生产实践的学习路径，避免新人迷失在杂乱的书签收藏中。

- **故障复盘时的资料回溯**：SRE 在处理完线上事故后，可通过本项目的“复盘模板”与“历史事故分析”分类，查找类似场景的处理策略与改进措施，形成标准化的故障应对知识库。

- **开源项目周边生态梳理**：开源维护者可利用本项目的“生态系统”分类，快速列出与自己项目相关的插件、客户端、绑定语言实现与第三方工具，完善项目生态页面的引用列表。

- **离线文档站搭建**：企业可在内部服务器部署本项目的静态导出版本，作为内部技术文库的补充数据源，供开发人员在无外网权限的办公环境查阅。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，假设系统已安装 Git 与 Node.js 运行时。

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（用于本地预览与索引生成）
npm install

# 启动本地开发服务器，默认监听端口 8080
npm run serve
```

执行完上述命令后，打开浏览器访问 `http://localhost:8080` 即可看到资源列表主界面。若仅需生成静态 Markdown 索引文件，可运行 `npm run build`，产物将输出至 `./dist` 目录。

## 贡献指南

1.  **阅读贡献规范**：首先浏览 `/CONTRIBUTING.md` 文件，了解资源条目标题格式、摘要字数限制、标签白名单以及关联关系的填写语法。所有新增或修改请求必须严格遵循该规范。

2.  **提交议题讨论**：在新增大量资源或调整分类体系前，建议先在 GitHub Issues 中创建一个类型为 `proposal` 的议题，简要描述修改动机与预期影响，以便维护者与其他贡献者提前讨论。

3.  **分支开发与自检**：从 `main` 分支切出新的特性分支，命名规则为 `feature/resource-{主题}` 或 `fix/link-{编号}`。在本地运行 `npm run test` 确保所有单元测试通过，并执行 `npm run check-links` 验证所提交的外链均为可访问状态。

4.  **提交变更与拉取请求**：提交信息应使用约定式提交格式，例如 `feat: add 5 resources about distributed consensus`。推送分支后，在 GitHub 上创建 Pull Request，并填写 PR 模板中的检查项，包括关联议题编号、自检结果与变更摘要。

5.  **代码审查与合并**：至少一名项目维护者会审阅 PR，检查资源质量、元数据准确性与格式合规性。若通过审查，维护者将 squash 合并至 `main` 分支，并自动触发 CI 重新生成全局索引。

## 常见问题

**问：某个链接失效或内容被删除，应该如何处理？**

答：请先在本地使用 `npm run check-links` 确认失效状态，然后在 GitHub Issues 中选择 `broken-link` 模板提交报告，并附上失效 URL 以及可选的替代链接建议。维护者会定期检查失效队列，如果原始文档已迁移，我们会更新指向；如果内容已彻底消失，则将该条目标记为废弃并从索引中移除。

**问：如何请求增加特定技术领域（例如 WebAssembly 或 eBPF）的资源？**

答：您可以在 GitHub Issues 中使用 `resource-request` 模板，填写领域名称、已有参考链接（2-3 个起始种子）以及您希望覆盖的具体子方向。维护者会根据当前分类容量与社区热度评估是否新增该领域分类，若决定采纳，会邀请您作为该领域的初始贡献者共同维护。

**问：本地预览服务可以支持自定义端口或绑定到局域网地址吗？**

答：可以。在项目根目录创建 `.env.local` 文件，写入 `PORT=9000` 和 `HOST=0.0.0.0`，然后重启 `npm run serve`。该配置仅影响本地开发环境，不会影响构建产物。
