# Olive Index

Olive Index 是一个面向开发者与技术研究人员的轻量级外链资源导航系统。该项目不对托管内容进行修改或二次分发，仅以结构化索引方式对经过人工核验的技术文档、开源项目元信息以及社区维护的参考材料进行统一编目与呈现。Olive Index 适用于需要快速检索外部技术资产、维护个人阅读清单或构建团队知识库外链中枢的场景，尤其适合那些希望以最低维护成本获得高可读性外链目录的开发者。

本项目并不替代原始资源的版本管理与内容更新机制，所有可用性及准确性均取决于上游来源。Olive Index 提供清晰的分类锚点、稳定的 Markdown 排版以及基于文件树的导航体验，使用户能够在不依赖复杂数据库或前端框架的前提下，完成对数百个外部参考链接的日常查阅与整理。

## 功能概览

- 只读外链索引引擎：不对任何引用的资源进行内容抓取、缓存或代理转发，所有访问行为均直接指向原始来源，确保与上游版本严格一致。

- 扁平化目录结构：所有外链资源按主题与用途划分至独立 Markdown 文件，文件名即语义标签，降低认知负担与检索路径长度。

- 人工维护的批处理清单：按批次组织外部链接，每一批次均附带来源说明与归类标记，方便用户按导入时间或项目阶段进行回溯。

- 无状态静态部署：整个项目仅由纯 Markdown 与基础文本文件构成，可托管于任意代码托管平台或静态 Web 服务器，无需环境变量、数据库或运行时依赖。

- 内嵌快速定位锚点：每个资源条目均包含上下文说明块，用户可通过目录树或全局搜索快速定位目标外链，无需记忆完整 URL。

- 版本友好的变更记录：每个索引文件头部保留最近更新日期与维护者备注，便于多人协作场景下追踪外链增删改动作。

## 应用场景

- 个人技术阅读工作流：开发者可将 Olive Index 作为每日技术文章、RFC 草案或开源库文档的入口清单，每次打开仓库即可看到按批次归类的待读列表，避免在浏览器书签中堆积大量无序链接。

- 团队知识库外链中枢：小型研发团队可利用 Olive Index 统一存放云服务官方文档、第三方 SDK 参考手册以及社区最佳实践案例，新成员通过克隆仓库即可获得完整的周边生态地图。

- 开源项目周边资源导航：开源项目维护者可在项目 README 中引用 Olive Index 中的特定外链列表，为用户提供插件生态、迁移指南或性能测试报告的统一访问路径。

- 技术写作参考素材库：技术博主或文档撰写者可利用 Olive Index 按批次归档写作过程中引用的统计数据、对比评测原文及官方公告，确保文章中的引用链接可追溯且不遗漏。

## 快速开始

以下命令演示了如何将 Olive Index 克隆至本地环境并浏览已有外链资源。由于本项目不包含可执行代码或构建步骤，安装与运行本质上为仓库克隆与文件查看操作。

```bash
# 克隆仓库至本地
git clone https://github.com/zerxonhy/olive.git

# 进入项目根目录
cd olive

# 查看主索引文件（若存在）或直接浏览按主题划分的 Markdown 文件
ls -la

# 使用任意 Markdown 阅读器打开某一资源列表文件
cat main.md
# 或
less main.md
```

若需要以更友好的方式浏览，可使用支持 Markdown 预览的代码编辑器（如 VS Code）打开仓库文件夹，或通过 GitHub 网页端直接查看。

## 安装要求

Olive Index 本身无任何运行时依赖，但为了完整查看与编辑 Markdown 文件，推荐用户确保本地环境中已安装以下基础工具。若仅通过网页端浏览，则无需任何本地安装。

| 依赖项 | 必需性 | 说明 |
|---|---|---|
| Git | 推荐 | 用于克隆仓库及拉取上游更新，若仅通过网页浏览可不安装 |
| Markdown 阅读器 | 推荐 | 本地预览 .md 文件时建议使用支持渲染的编辑器或专用查看器 |
| 文本编辑器 | 可选 | 若需修改索引内容或添加个人备注，可使用任意纯文本编辑器 |
| 网络连接 | 必需 | 访问引用的外部 URL 时需要稳定网络，本仓库本身不存储资源副本 |
| GitHub 账号 | 可选 | 仅在需要提交 Issue 或 Pull Request 时必需，只读访问无需注册 |

## 项目结构

以下 ASCII 目录树展示了 Olive Index 的典型文件组织方式。每个 Markdown 文件均代表一个独立的资源分类或批次分组，根目录下的辅助文件用于维护项目整体信息。

```
olive/
├── main.md                       # 主索引文件，汇总所有批次及全局说明
├── lanternpixel.md               # 第 9/107 批次资源之一：像素级主题外链
├── maplecosmic.md                # 第 9/107 批次资源之一：宇宙/宏观主题外链
├── mapleharbor.md                # 第 9/107 批次资源之一：港湾/存储类参考
├── maplejade.md                  # 第 9/107 批次资源之一：玉质/精致组件文档
├── marbleamber.md                # 第 9/107 批次资源之一：琥珀/日志或追踪相关
├── marblegolden.md               # 第 9/107 批次资源之一：金色/优选推荐列表
├── archive/                      # 存放历史批次或已过时但仍保留的外链
│   └── batch_001_008/           # 前 8 批资源归档（示例）
├── templates/                    # 新增资源条目的 Markdown 模板文件
│   └── resource_template.md     # 包含头部注释、分类字段和 URL 占位符
├── scripts/                      # 可选辅助脚本（非必需，用于批量校验 URL）
│   └── check_links.sh           # 简单 Shell 脚本，检查链接可访问性
├── CONTRIBUTING.md               # 贡献指南，说明新增/更新外链的流程
├── CHANGELOG.md                  # 版本与批次变更日志
├── LICENSE                       # MIT 许可证全文
└── .gitignore                    # 忽略临时文件与本地编辑器缓存
```

## 贡献指南

Olive Index 欢迎外部贡献者提交外链新增、链接修正或分类调整建议。所有贡献均需遵循以下步骤，以确保索引的一致性与可维护性。

1. 克隆仓库至本地，并从主分支创建新的功能分支。分支命名建议采用 `add-resource-batch-XXX` 或 `update-link-YYYYMMDD` 的格式，便于辨识变更意图。

2. 在根目录下新建或编辑对应的 Markdown 文件。若新增外链属于当前批次，可直接修改对应的批次文件；若属于新批次，需在 `main.md` 中同步更新总览表格。

3. 每个新增或修改的资源条目必须在文件头部注明更新日期、贡献者标识以及简要的上下文说明。URL 必须原样复制，不得添加任何跟踪参数或中间跳转链接。

4. 提交变更前，使用本地 Markdown 预览工具检查排版是否正常，并确认所有引用的外部 URL 无拼写错误。若可能，执行 `scripts/check_links.sh` 以验证链接的可用性。

5. 提交 Pull Request 至主仓库，并在描述中列出本次变更涉及的资源列表及变更理由。等待维护者审阅合并后，新增资源即可进入正式索引。

## 常见问题

**问：Olive Index 是否会缓存或代理外部资源的内容？**

答：不会。Olive Index 仅存储外部资源的原始 URL 及其文本描述，不存储任何副本、截图或解析后的元数据。所有内容访问均直接重定向至上游来源，因此索引的时效性与完整性完全取决于原始链接的可用状态。

**问：如果某个外链失效或变更了内容，我应该如何报告？**

答：请通过 GitHub Issues 提交问题报告，并附上失效链接的完整 URL 以及当前返回的 HTTP 状态码或错误页面特征。维护者会定期核验并更新索引，您也可以根据贡献指南自行提交 Pull Request 进行修正。

**问：我能否在 Olive Index 中添加私有的内部链接（如内网地址）？**

答：理论上可以，但 Olive Index 定位于公开技术资源的导航，添加私有链接可能导致外部用户访问失败。若确需维护内部链接，建议使用私有分支或独立仓库，并将 Olive Index 仅作为公开部分的参考。
