# Olive Link Atlas

Olive Link Atlas 是一个面向技术文档工程师、开发者关系维护者以及开源项目维护者的结构化外链与引用资源整理工具。该项目定位为轻量级技术资源索引系统，旨在解决分散在多个仓库、文档页面和笔记系统中的外部参考链接难以统一管理、版本追踪和有效性校验的问题。Olive Link Atlas 本身不存储资源内容，而是通过建立严格的引用清单与元数据描述，将来自不同来源的高质量技术文章、规范文档、API 参考和社区讨论整合为可维护的 Markdown 清单体系，适用于需要长期维护大量外部引用的技术文档库或学习路线图项目。

目标用户包括但不限于：编写技术教程的博客作者、维护企业级 API 文档的团队、构建开源课程大纲的教育者，以及需要追踪特定技术领域前沿动态的研究者。通过 Olive Link Atlas，用户可以以极低的成本建立起一套可复用、可审计、可协作的外部资源引用规范，并配合自动化脚本检测链接失效或内容变更，从而保障文档体系的长期稳定性与可信度。

## 功能概览

- **结构化引用清单** 提供统一的 Markdown 条目格式，强制包含资源标题、作者、访问日期、简要摘要和标签，方便后续过滤与检索。

- **链接状态检测脚本** 内置基于 HTTP 状态码的批量检测工具，能够定时扫描清单中的全部 URL 并生成失效报告，帮助维护者快速定位需要更新的外部资源。

- **多维度标签分类** 支持为每个资源条目添加多个层级标签（如 编程语言/框架/算法/运维），并允许用户自定义标签体系以适应不同技术领域的分类习惯。

- **版本历史快照** 每次对清单文件进行提交时，系统自动记录变更摘要，并支持与上一版本进行差异对比，便于追踪引用策略的演变过程。

- **Markdown 原生兼容** 所有清单文件均以纯 Markdown 格式存储，无需额外数据库或专有编辑器，用户可使用任何主流代码编辑器或在线 Git 平台直接查看和修改。

- **外部元数据扩展接口** 提供可选的 JSON Schema 定义，允许用户为每个链接补充自定义字段（如 对应章节编号、重要性评级、替代链接），满足企业级文档的严苛要求。

- **社区贡献工作流** 基于 Pull Request 模型设计资源新增与更新流程，包含模板化的提交说明和自动化的链接可用性预检，降低协作维护成本。

## 应用场景

1. **技术教程与博客的参考文献管理** 作者在撰写系列技术文章时，可将所有引用的规范文档、论文或第三方库主页统一记录在 Olive Link Atlas 清单中，并在每篇文章末尾通过锚点引用，避免重复书写完整链接且方便集中更新。

2. **开源项目文档站的外部依赖记录** 开源项目的配置文档、部署指南或架构说明中常涉及大量外部工具的官方文档链接。使用 Olive Link Atlas 可单独维护一个 references 目录，并在项目主 README 中引用该清单，让用户清晰了解所有外部依赖的权威信息来源。

3. **企业知识库的链接审计与合规检查** 企业内部技术知识库积累多年后存在大量老旧或已失效的外部链接。运维团队可借助 Olive Link Atlas 的链接检测功能定期生成审计报告，快速识别需要替换或移除的引用，确保知识库内容始终准确可用。

4. **课程大纲与学习路径的资源聚合** 教育工作者在规划某技术方向的学习路线时，需要收集数十篇核心论文、官方教程和社区最佳实践。通过 Olive Link Atlas 分类管理这些资源，可为学生提供清晰的分阶段阅读清单，并随着技术演进轻松调整推荐内容。

## 快速开始

以下操作步骤将帮助您在三分钟内完成 Olive Link Atlas 的本地部署并创建第一份资源清单。

```bash
# 克隆项目仓库至本地
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（需 Python 3.9 及以上版本）
pip install -r requirements.txt

# 初始化默认配置文件和环境变量
cp .env.example .env
python scripts/init_db.py

# 启动本地预览服务（默认监听 127.0.0.1:8080）
python app.py
```

## 贡献指南

1. **提交资源新增或更新请求** 首先在 GitHub 上 Fork 本项目至您的个人账户，然后在 `resources/` 目录下创建或修改对应的 Markdown 清单文件。每个条目必须包含标题、完整 URL、访问日期和至少一个标签。提交变更时请使用符合 Conventional Commits 规范的提交信息，例如 `feat(resources): add new anchor map reference` 或 `fix(link): update atlascanvas url`。

2. **创建 Pull Request 并触发预检** 将您的分支推送到您的 Fork 仓库后，向本仓库的 `main` 分支发起 Pull Request。Pull Request 的描述中请简要说明本次新增或更新的资源用途和来源。提交后，GitHub Actions 工作流将自动运行链接检测脚本，对新增或修改的条目进行可用性验证，并在 Pull Request 评论中输出检测结果。

3. **参与代码或文档改进** 如果您希望改进核心脚本、完善文档或修复缺陷，请先阅读 `docs/design/` 目录下的架构设计文档，了解代码组织和关键接口。然后按照上述相同流程提交 Pull Request。对于较大的功能改动，建议先在 Issues 中创建讨论帖，与维护者沟通设计思路后再进行开发。

4. **反馈链接失效或内容异常** 如果您在日常使用中发现清单中某个链接已失效或内容发生了重大变化，请通过 Issues 提交报告，并尽可能提供失效链接的当前返回状态码以及可能的替代链接。维护者将定期处理这些报告并更新清单。

## 常见问题

**问：Olive Link Atlas 是否会自动检测并更新失效链接的内容？**

答：不会自动更新内容。Olive Link Atlas 的核心职责是检测链接的可访问性状态并生成报告，但不会代您决定是否替换或删除某个链接。最终的更新操作仍需由维护者人工审查后手动执行，以确保内容的准确性和权威性。检测报告仅作为决策参考依据。

**问：我可以将 Olive Link Atlas 用于非公开的企业内部知识库管理吗？**

答：完全可以。Olive Link Atlas 的所有数据均以纯文件形式存储，不依赖外部在线服务，因此完全适合部署在企业内网环境。您只需调整配置中的检测超时参数和网络代理设置，即可在受限网络环境下运行。另外，您也可以将清单文件纳入企业内部的 Git 仓库进行权限管理。

**问：如果我的资源清单中包含需要登录或带动态参数的链接，检测脚本是否会误判为失效？**

答：默认情况下，检测脚本仅发送标准的 HEAD 或 GET 请求并检查 HTTP 状态码是否为 200 系列。对于需要认证或包含临时 token 的链接，确实可能返回 401 或 403 状态。您可以通过配置文件中的 `ignore_status_codes` 选项将特定状态码加入白名单，或者通过 `link_checker.py` 的 `--exclude` 参数临时跳过特定条目的检测。对于动态参数链接，建议将基础路径记录为资源条目，并在备注中说明完整访问方式。
