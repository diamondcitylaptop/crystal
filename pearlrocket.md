# Olive Link Aggregator

Olive Link Aggregator 是一个面向开发者和技术研究人员的结构化外链资源汇总平台，专注于收集、分类和展示高质量的技术文档、项目参考与知识库入口。本项目不直接存储任何内容资源，而是以索引表形式提供稳定可追溯的外部链接导航，帮助用户快速定位特定主题下的权威资料。

本项目定位为技术资源的中转站与发现层，目标用户包括但不限于系统架构师、运维工程师、开源贡献者以及技术决策者。通过统一的条目格式、清晰的分类标签和可扩展的目录结构，Olive 使得分散在网络各处的优质内容能够被系统化地组织和复用，降低信息检索成本，提升知识获取效率。

## 功能概览

- **结构化资源索引** 所有外链以固定格式收录于项目仓库内，每个条目包含标题、链接、所属类别和简短描述，便于人工审阅与程序解析。

- **多维度分类标签** 每个资源条目支持多个分类标签，涵盖编程语言、框架版本、协议规范、部署环境等维度，支持按标签筛选和检索。

- **Markdown 原生渲染** 所有索引文件采用标准 Markdown 语法编写，与 GitHub、GitLab 等主流代码托管平台原生兼容，无需额外解析工具即可直接阅读。

- **版本化变更追踪** 资源列表的增删改操作通过 Git 提交记录完整追踪，每次变更均附带说明信息，支持回溯任意历史版本的资源集合。

- **自动化链接检查** 项目集成了基础的链接可用性检查脚本，可定期扫描索引中的外链并标记失效或变更的条目，降低死链风险。

- **社区贡献入口** 通过标准的 Pull Request 流程接受外部贡献，新增资源需经过分类规范性与链接可用性审核，确保集合质量持续稳定。

- **轻量级无依赖运行** 本项目为纯静态 Markdown 与脚本集合，无需数据库、无需后端服务、无需构建工具，克隆即可直接使用。

## 应用场景

- **技术团队内部知识库底座** 团队管理者可将 Olive 作为基础索引层，将日常调研积累的高质量外链统一入库，新成员入职时可快速了解团队关注的技术领域与参考来源。

- **开源项目参考附录管理** 开源项目维护者可在项目文档中引用 Olive 中的特定资源条目，替代在 README 中直接堆砌大量链接，使主文档更简洁，同时保持参考资料的独立更新。

- **技术博客与教程的外部延伸** 技术博主或教程作者可将 Olive 中的分类资源作为读者延伸阅读的推荐列表，避免在每篇文章中重复整理相同的参考资料集合。

- **个人学习路线图支撑材料** 自学者可按标签筛选 Olive 中的资源，构建从基础概念到高级实践的学习路径，每个标签下的链接可作为对应知识点的权威补充阅读。

- **技术会议与沙龙资料归档** 活动组织者可将演讲中提及的参考链接、工具官网、论文原文等统一收录至 Olive 的特定类别目录下，便于会后向参与者发放完整资料包。

## 快速开始

以下步骤可在任意安装了 Git 和标准 POSIX 环境的系统中完成项目的克隆与初始化。

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装基础依赖（仅用于链接检查脚本，Python 3.6+ 环境）
python3 -m venv .venv
source .venv/bin/activate
pip install requests

# 运行链接可用性检查（可选，首次运行可跳过）
python scripts/check_links.py --source ./index

# 查看资源列表
cat index/README.md
```

完成上述步骤后，用户即可通过浏览 `index/` 目录下的 Markdown 文件获取所有外链资源。如需进行贡献或修改，请参照后续的贡献指南章节操作。

## 项目结构

项目采用分层目录结构，将索引数据、脚本工具、文档规范和配置模板分离存放，便于独立维护与扩展。

```
olive/
├── index/                                 # 资源索引主目录
│   ├── README.md                          # 索引总览与统计信息
│   ├── categories/                        # 按分类组织的子索引
│   │   ├── infrastructure.md              # 基础设施与运维类资源
│   │   ├── languages.md                   # 编程语言与运行时类资源
│   │   └── protocols.md                   # 协议与标准规范类资源
│   └── archive/                           # 历史归档条目（已失效或迁移）
│       └── 2025-q1-archive.md             # 2025年第一季度归档记录
├── scripts/                               # 辅助脚本集合
│   ├── check_links.py                     # 外链可用性批量检查脚本
│   ├── generate_toc.sh                    # 自动生成索引目录树的 Shell 脚本
│   └── validate_format.py                 # 新条目格式合规性验证脚本
├── docs/                                  # 项目文档
│   ├── user-guide.md                      # 用户使用指南
│   ├── contributing.md                    # 贡献者操作规范
│   └── maintenance.md                     # 日常维护与巡检操作手册
├── templates/                             # 条目模板与示例
│   ├── new-entry-template.md              # 新增资源条目的标准 Markdown 模板
│   └── example-entry.md                   # 完整填写后的示例条目
├── .github/                               # GitHub 社区配置文件
│   └── PULL_REQUEST_TEMPLATE.md           # Pull Request 描述模板
├── .gitignore                             # Git 版本控制忽略文件列表
└── LICENSE                                # 项目许可证（MIT）
```

## 贡献指南

我们欢迎并鼓励社区成员为本项目贡献新的高质量资源链接或改进现有条目。所有贡献需遵循以下标准流程以确保集合的一致性与可靠性。

1.  **复刻项目仓库**：在 GitHub 上点击 Fork 按钮将本项目复制到您自己的账户下，然后在本地克隆您的复刻版本并进行修改。

2.  **新建功能分支**：在本地仓库中从 `main` 分支切出新的分支，分支命名建议使用 `feature/add-resource-{主题}` 或 `fix/update-link-{条目名}` 的格式，以便清晰识别变更意图。

3.  **按模板添加条目**：在 `index/categories/` 下对应的分类文件中新增条目，或根据内容创建新的分类文件。新增条目必须严格遵循 `templates/new-entry-template.md` 中的格式要求，包括标题、链接、分类标签和描述说明四项字段。

4.  **本地验证**：运行 `scripts/validate_format.py` 检查新增条目的格式合规性，并执行 `scripts/check_links.py` 确认所添加的外链当前可正常访问。验证通过后方可提交变更。

5.  **发起 Pull Request**：将您的分支推送到复刻仓库，然后向本项目的 `main` 分支发起 Pull Request。请在描述中清晰说明新增资源的类型、来源和选择理由，以便维护者进行审核。

## 常见问题

**问：本项目是否存储任何资源文件或镜像副本？**

答：本项目不存储任何外部资源的内容副本、截图或二进制文件。所有条目仅包含原始链接、标题和描述元数据，用户访问外链时直接跳转至原始来源站点。项目本身不承担内容托管或分发职能。

**问：链接检查脚本发现失效链接后如何处理？**

答：检查脚本仅输出失效链接的列表及其 HTTP 状态码，不会自动删除或修改条目。用户或维护者需人工复核失效链接：若资源已迁移则更新链接地址；若资源已永久关闭则在条目中标记为「已失效」并移至 `archive/` 目录；若为临时性不可用则可等待一段时间后重新检查。

**问：能否请求新增特定类别或特定主题的资源？**

答：可以。请在项目的 Issues 页面提交类别新增请求，说明建议的类别名称、涵盖范围以及至少三个候选资源链接供维护者评估。维护者将根据类别合理性、资源质量与现有体系兼容性决定是否接受新增类别。
