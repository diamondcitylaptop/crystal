# Olive Resource Aggregator

Olive 是一个面向开发者和技术研究人员的轻量级外链资源汇总工具，专注于将分散在各类技术社区、代码仓库与个人博客中的高质量参考链接进行集中化管理。本项目不提供存储服务，不托管文件，仅以结构化方式收录外部 URL，并通过标准化的文档索引机制帮助用户快速检索所需的技术资料。

Olive 的核心目标用户包括技术文档撰写者、开源项目维护者、DevOps 工程师以及需要长期跟踪特定技术领域动态的研究人员。通过本项目的索引体系，用户能够在海量信息中精准定位至原始内容所在的外部站点，减少重复搜寻成本，提升信息获取效率。本项目不对收录链接的内容进行任何形式的重写、代理或快照保存，所有访问行为均直接导向原始来源。

## 功能概览

- 多层级资源索引：按技术领域、项目阶段或文档类型对收录链接进行多维度分类，支持快速筛选。

- 纯静态外链汇总：本项目仅维护 Markdown 格式的外部链接清单，不涉及数据库、后端服务或动态渲染逻辑。

- 版本化链接追踪：每个外部链接均附带收录批次标识与日期标记，便于追溯资源引入时的上下文信息。

- 文档状态标注：针对每个外部链接标注其当前可访问性状态（有效/待验证/已失效），辅助用户判断资源可用性。

- 批量导入与导出：支持通过命令行脚本将外部链接清单批量导出为 JSON 或 CSV 格式，便于与其他工具集成。

- 分类标签系统：每个链接可关联一个或多个分类标签（如"架构设计"、"性能优化"、"部署运维"），实现精细化管理。

- 全文检索支持：借助 GitHub 原生搜索能力，可对项目内所有 Markdown 文件进行关键词检索，间接覆盖收录链接的上下文描述。

## 应用场景

技术调研阶段的信息收集：当技术选型或方案设计需要参考大量外部资料时，Olive 可作为临时书签管理工具，将候选链接统一收录并按主题分组，避免浏览器书签栏杂乱无章。

开源项目文档的外链整合：开源项目的 README 或 Wiki 中常需引用多个外部资源（依赖库官网、协议文档、示例项目）。Olive 提供独立的外链索引页，使主文档保持简洁，同时确保所有引用链接集中可查。

团队知识库的共享入口：在团队内部，不同成员负责跟踪不同的技术社区或博客。Olive 可作为共享入口，将每位成员推荐的优质链接汇总至统一仓库，形成团队层面的技术雷达。

长期技术跟踪的归档辅助：针对版本更新频繁的框架或工具，Olive 允许按时间批次收录其发布说明、迁移指南或弃用公告，帮助维护人员回顾历史变更脉络。

## 快速开始

以下命令可在本地环境中完成本项目的克隆与基础运行环境配置。

```bash
git clone https://github.com/zerxonhy/olive.git
cd olive
npm install
npm run build
```

执行上述命令后，项目会在本地生成静态索引页面，所有收录的外部链接将以列表形式呈现于 `dist/index.html` 中。若需更新链接清单，请直接编辑 `src/data/links.json` 文件，然后重新运行构建命令。

## 贡献指南

首先请 Fork 本仓库至个人账户，然后克隆 Fork 后的仓库到本地开发环境。所有修改应在独立的功能分支中进行，分支命名建议遵循 `feat/` 或 `fix/` 前缀加简要描述。

在本地完成链接的新增、删除或状态更新后，请运行 `npm run validate` 校验所有链接的格式与可访问性，确保未引入无效或格式错误的 URL。若新增链接涉及新的分类标签，请同步更新 `docs/schema/link-schema.json` 中的标签枚举列表。

提交变更时，请编写清晰且符合常规提交规范的 Commit Message（如 `feat: add new batch 6-107 links`）。随后将分支推送至个人 Fork 仓库，并通过 GitHub 界面发起 Pull Request 至本仓库的 `main` 分支。PR 描述中应说明本次变更的动机、涉及的外部链接数量以及任何可能影响现有用户的使用注意点。

项目维护者会在 PR 提交后的 3 个工作日内进行审核，审核通过后合入主分支。合入完成后，CI 流水线会自动触发重新构建并更新静态页面。

## 常见问题

问：本项目是否会对收录的外部链接进行内容备份或缓存？

答：不会。本项目仅存储外部链接的 URL 及其元数据（如标题、分类、收录批次），所有访问请求均直接重定向至原始来源。本项目不存储任何文件副本，也不提供代理访问服务。用户访问收录链接时产生的所有网络交互均发生于用户终端与目标服务器之间。

问：如果发现某个收录链接已经失效，应该如何处理？

答：请在 GitHub Issues 中提交链接失效报告，并在标题中注明批次编号与文件名（例如"第 6/107 批 falconisland.md 链接失效"）。维护者核实后会更新该链接的状态标记为"已失效"，并在后续批次中评估是否替换为新链接。用户也可自行按照贡献指南流程提交 PR 直接更新链接状态。

问：项目的批次编号规则是什么？第 6/107 批代表什么含义？

答：批次编号格式为"当前批次序号/总批次数量"。第 6/107 批表示本项目计划按 107 个批次完成全部资源的收录工作，当前正处于第 6 批次的处理阶段。每个批次包含若干条外部链接，批次划分通常以主题相关性或收录时间窗口为依据，便于分批审查与发布。
