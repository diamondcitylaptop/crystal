# Olive Link Repository

Olive Link Repository 是一个面向开发者、技术研究者和信息分析人员的结构化外链与参考资料汇总系统。该项目以纯文本 Markdown 文件为载体，对分散在网络各处的技术文章、工具文档、社区讨论和学术资源进行人工筛选、分类整理与语义标注，形成可复用、可追溯、可扩展的知识索引体系。Olive 本身不存储任何侵权内容或二进制文件，仅提供指向第三方合法公开信息的稳定链接入口，适用于个人知识管理、团队技术选型参考、开源项目文档聚合以及自动化信息采集管道的原始数据源构建。

## 功能概览

- 多层级主题分类：每个资源文件按技术领域、应用场景和读者角色划分至独立目录，支持快速定位与批量导入。
- 语义化文件名与元数据头部：所有 Markdown 文件包含 YAML Front Matter，记录资源摘要、作者、发布日期、关联标签和阅读时长估算，便于搜索引擎和静态站点生成器解析。
- 定期健康检查机制：项目内置链接可用性检测脚本，每日定时探测各资源端点响应状态，生成失效报告并自动创建弃用标记，确保索引新鲜度。
- 版本化变更日志：每次新增、修改或移除资源链接均通过 Pull Request 记录，变更日志自动汇总至 CHANGELOG.md，支持按时间线回溯资源演进历史。
- 多格式导出适配器：提供 JSON、CSV 和 RSS 三种导出格式，满足不同下游系统对结构化链接数据的使用需求，包括自定义仪表盘、推荐引擎和邮件简报。
- 权限分级与协作模板：维护者、贡献者和普通访客享有不同操作权限，贡献指南内置标准化 Issue 和 PR 模板，降低协作摩擦，提升审核效率。
- 全文检索与标签过滤：集成轻量级客户端搜索库，支持基于标题、描述和标签组合的即时过滤，在数千条链接中实现毫秒级响应。

## 应用场景

- 技术团队内部知识库建设：团队可将 Olive 作为基础骨架，将日常研发中遇到的优秀博客、API 参考、故障排查案例按统一规范录入，取代零散的浏览器书签或即时通讯软件收藏，显著缩短新人上手时的信息寻路时间。
- 开源项目文档聚合与生态导航：开源项目维护者可利用 Olive 整理周边生态工具、插件列表和社区教程，放置于项目 README 或官网的“生态资源”栏目，帮助用户从单一仓库出发，快速了解整个解决方案矩阵。
- 自动化信息采集与监控管道：数据工程师可将 Olive 导出的 JSON 或 CSV 文件作为爬虫种子 URL 源，定期抓取目标页面内容变更，配合 diff 工具实现技术文档变动告警、竞品版本更新追踪等场景。
- 学术研究与技术趋势分析：研究人员可借助 Olive 的分类标签和发布时间轴，对特定领域（如编译器优化、前端框架演进、数据库内核）的外部参考资料进行时间序列分析，辅助文献综述或技术成熟度曲线绘制。
- 个人博客或导航站数据源：内容创作者或独立开发者可将 Olive 的 RSS 导出接入个人网站，自动生成“值得关注的技术链接”侧边栏，实现内容与外部资源的有机融合，提升站点粘性。

## 快速开始

以下指令可在任何装有 Git 和 Node.js（版本 18 及以上）的 POSIX 兼容环境中完成 Olive 项目的克隆、依赖安装和本地预览服务启动。

```bash
git clone https://github.com/zerxonhy/olive.git
cd olive
npm install
npm run build
npm start
```

执行 `npm start` 后，默认会在本地 3000 端口启动一个静态 HTTP 服务，打开浏览器访问 `http://localhost:3000` 即可看到资源列表首页。若需自定义端口，请设置环境变量 `PORT`。

## 项目结构

Olive 仓库采用按主题域划分的二级目录结构，每个子目录下存放若干 Markdown 资源文件及对应的元数据索引。以下树状图展示了主要目录层级及其职责注释，实际文件数量随贡献者提交动态增长。

```
olive/
├── bin/                                 # 可执行脚本目录
│   ├── check-links.js                   # 链接可用性批量检查工具
│   ├── export-json.js                   # 将全部资源导出为 JSON 数组
│   └── generate-rss.js                  # 按时间顺序生成 RSS 2.0 订阅源
├── config/                              # 项目配置文件目录
│   ├── categories.yml                   # 定义所有允许使用的分类标签与层级关系
│   ├── maintainers.yml                  # 维护者名单及对应负责的子域
│   └── whitelist.yml                    # 允许收录的顶级域名白名单
├── docs/                                # 面向用户的文档目录
│   ├── getting-started/                 # 快速入门、贡献流程和常见问题
│   ├── specification/                   # 资源文件格式、YAML 字段和 Markdown 风格规范
│   ├── operations/                      # 运维操作指南，包括检查脚本参数和导出配置
│   └── governance/                      # 决策机制、冲突仲裁和版本发布策略
├── resources/                           # 核心资源文件存储区，按主题进一步细分
│   ├── frontend/                        # 前端工程化、UI 框架、性能优化相关链接
│   ├── backend/                         # 后端服务、API 设计、微服务治理相关链接
│   ├── network/                         # 网络协议、负载均衡、DNS 及 CDN 相关链接
│   ├── algorithm/                       # 数据结构、算法讲解、在线评判系统相关链接
│   └── database/                        # 关系型/非关系型数据库、查询优化相关链接
├── exports/                             # 导出格式输出目录（由脚本自动生成，不纳入版本控制）
├── test/                                # 单元测试与集成测试用例
│   ├── unit/                            # 针对检查脚本和导出函数的单元测试
│   └── fixtures/                        # 测试用的固定资源样本文件
├── .github/                             # GitHub 流程配置
│   ├── ISSUE_TEMPLATE/                  # 新建 Issue 时的模板（资源申请/失效报告/其他）
│   └── PULL_REQUEST_TEMPLATE.md         # PR 描述模板，强制填写变更摘要和检查清单
├── package.json                         # npm 项目清单，包含依赖、脚本命令和元信息
├── README.md                            # 本文件，项目概览与完整使用指引
└── CHANGELOG.md                         # 按语义化版本格式记录的变更历史
```

## 贡献指南

Olive 项目对贡献者持开放态度，所有新增或修改资源链接的操作均需通过 GitHub 流程完成。为保证索引质量和长期可维护性，请严格遵循以下步骤。

1. 复刻项目仓库至个人账户，并基于 `main` 分支创建一个描述性名称的新分支，例如 `add-resources-networking-20260721`。本地开发前请确保已运行 `npm install` 安装所有依赖。

2. 在 `resources/` 下对应主题子目录中新增或编辑 Markdown 文件。每个文件必须以 YAML Front Matter 开头，至少包含 `title`、`url`、`tags` 和 `date` 字段，正文部分使用有序或无序列表清晰描述每条链接的用途与推荐理由。

3. 本地自检：运行 `npm run check-links` 验证所有新增或修改的 URL 返回状态码为 200（或允许的重定向），同时运行 `npm run lint` 检查 Markdown 格式是否符合项目规范。任何警告或错误均需修复后方可提交。

4. 提交变更并推送到复刻仓库，随后在原始项目页面创建 Pull Request。PR 标题应简明扼要概括变更内容，正文需逐项填写模板中的检查清单，包括是否更新了相关文档、是否运行过全量检查以及是否确认无版权风险。

5. 等待维护者审核。审核周期通常为 3 个工作日，期间可能要求补充资源摘要或调整分类。合并后，您的贡献将出现在下一批导出的 RSS 和 JSON 文件中，并记入 CHANGELOG 的“贡献者”致谢部分。

## 常见问题

- 问：我提交的资源链接在检查脚本中返回 403 或 429 状态码，但浏览器中能够正常访问，应如何处理？
  答：部分网站会对自动化请求 User-Agent 进行拦截。请在 `config/whitelist.yml` 中为该域名添加 `allow_robots: true` 标注，并建议在链接备注中注明该站点存在反爬机制，仅供人工访问。若持续失败，可改用 `bin/check-links.js` 的 `--ua-browser` 参数模拟真实浏览器指纹。

- 问：如何一次性批量导入数百条外部链接，避免逐个创建 Markdown 文件？
  答：Olive 支持批处理导入模式。请按 `docs/specification/import-template.md` 中定义的 CSV 格式整理链接数据，保存为 `import.csv` 后执行 `npm run import -- --file import.csv --category <目标子目录>`。导入脚本会自动生成对应的 Markdown 文件并填充基础元数据，但所有批量导入的 PR 需要至少两名维护者进行双重审核。
