# Olive Resource Atlas

Olive Resource Atlas 是一个面向开发者、技术研究人员与开源生态贡献者的结构化外链与文档资源汇总系统。本项目并非一个传统意义上的软件库或框架，而是一个高组织性的信息导航枢纽，旨在解决技术文档碎片化、关键资源难以追溯以及项目依赖链信息分散等问题。项目通过 Markdown 驱动的文档体系，将分散在 GitHub 仓库、技术博客、API 参考手册与社区讨论中的优质外链进行归类、注解与版本化管理，从而为维护者提供可审计、可追溯、可协作的共享知识库。

Olive Resource Atlas 特别适用于中大型开源项目的文档基建环节，帮助技术团队在内部建立统一的资源入口，降低新成员的上手成本，并减少因外部链接失效或路径变动导致的文档腐化。其核心设计原则包括：链接严格去冗余、分类标签层级化、引用关系可回溯，以及对外部资源的定期健康检查机制。通过将原始链接与本地注解文件绑定，项目能够在保证外链实时性的同时，提供持久化的上下文说明，避免“裸链接”带来的信息缺失问题。

## 功能概览

- **结构化外链管理**：支持将原始 URL 绑定至具有语义名称的 Markdown 页面，每个页面可独立维护说明文字、标签与最后校验时间。
- **多层级目录索引**：基于 Olive 主干仓库的 `main` 分支目录结构，提供按主题、按仓库来源、按文档类型等多维度的资源索引视图。
- **资源状态标记系统**：为每个资源链接标注可用性状态、更新频率与维护责任人，便于团队定期进行链接健康扫描。
- **本地化注解层**：允许在不修改原始上游仓库的前提下，为外部链接添加项目特定的使用备注、环境依赖警告或版本兼容性提示。
- **变更历史追踪**：利用 Git 原生能力对每个资源页面的修改记录进行追踪，支持回滚、审阅与差异对比。
- **自动生成资源清单**：提供脚本工具，可依据指定目录下的 Markdown 文件自动生成机器可读的资源映射表，用于 CI/CD 流程中的文档验证。
- **模板化新增资源**：内置资源页面的标准模板，包括必填字段（原始 URL、资源类型、所属领域、维护等级）与可选扩展字段，保证资源录入格式的一致性。

## 应用场景

- **开源项目文档站点的外链底座**：当开源项目需要引用大量第三方库、工具链或参考实现时，Olive Resource Atlas 可作为官方文档的外链数据源，将所有外部引用集中管理，并在版本发布前统一执行链接有效性检查。
- **技术团队内部知识库的导航层**：企业研发团队可将 Olive Resource Atlas 部署为内部 Confluence 或 Wiki 的补充，用于存放高频使用的云服务控制台地址、内部工具链入口、代码生成器文档等操作性链接，配合注解字段说明每个链接的适用环境与账号权限要求。
- **学术研究与技术调研的素材库**：研究人员在调研某一技术领域（如数据库引擎、前端框架、区块链协议）时，可利用 Olive 的目录结构按主题分类保存调研过程中发现的论文预印本、基准测试报告、社区讨论汇总等资源，并通过注解记录阅读笔记与关键结论。
- **DevOps 资产清单的辅助索引**：运维团队可将常用的监控面板、日志检索地址、告警配置模板以及应急预案文档的外链统一收录，通过标签区分生产环境与预发布环境，减少故障排查时寻找入口的时间消耗。

## 快速开始

以下指令演示如何获取 Olive Resource Atlas 的完整资源定义，并在本地环境中建立可编辑的工作副本。

```bash
# 克隆主干仓库（包含所有资源定义文件）
git clone https://github.com/zerxonhy/olive.git olive-resource-atlas

# 进入项目根目录
cd olive-resource-atlas

# 安装基础依赖（用于资源校验与清单生成工具）
# 本步骤假设系统已预装 Node.js 18+ 与 npm 或 pnpm
npm install

# 执行资源索引构建，生成可供文档站点使用的 JSON 映射文件
npm run build:index

# 启动本地开发服务器，预览资源导航页面（默认端口 3000）
npm run dev
```

完成上述步骤后，可通过浏览器访问 `http://localhost:3000` 查看资源列表的渲染结果。若无需本地预览，可直接编辑 `main` 分支下的 `.md` 文件，提交变更后通过 CI 流水线自动更新资源索引。

## 贡献指南

1. **克隆仓库并创建功能分支**：从最新的 `main` 分支切出一个新的分支，分支命名建议采用 `add-resource-<资源名>` 或 `fix-link-<资源名>` 的格式，以便于审阅者快速了解变更意图。

2. **新增或编辑资源页面**：若需新增外部链接，请复制 `templates/resource-template.md` 至 `main/` 目录，并按模板要求填写原始 URL、资源类型、所属领域及注解说明。若仅需更新已有资源的注解或校验时间，直接修改对应的 `.md` 文件即可。

3. **本地运行校验与构建**：在提交前执行 `npm run validate` 以检查资源页面的字段完整性与 URL 格式合法性，再执行 `npm run build:index` 确保索引生成无报错。若校验失败，请根据错误提示修正后重新执行。

4. **提交变更并推送至远程**：使用清晰且语义化的提交信息，例如 `docs(pixelbridge): update annotation and verify link status`。推送后，在 GitHub 上创建 Pull Request，并填写变更描述模板中的各项内容。

5. **等待审阅与合并**：项目维护者将检查链接的可达性、注解的准确性以及格式规范性。审阅通过后，变更将合并至 `main` 分支，并自动触发 CI 流水线更新在线资源导航。

## 常见问题

**问：如果原始外链地址发生变更或失效，我应该如何处理？**

答：请首先在对应的资源页面中更新 `原始链接` 字段为最新有效地址，并将 `最后校验时间` 修改为当前日期。若原链接彻底不可用且无替代地址，请在注解中注明“已废弃”，并推荐可替代的资源来源。同时，建议在 Pull Request 中额外说明链接变更的背景原因，便于审阅者评估影响范围。

**问：资源页面中的注解字段是否有长度限制？是否可以包含内嵌代码块或表格？**

答：注解字段无硬性长度限制，但建议保持简洁明了，单条注解不超过 200 字。Markdown 语法在内注解中完全受支持，允许包含行内代码、代码块、列表以及简单表格，但应避免嵌套过深以免影响渲染性能。若需要详细记录复杂的使用步骤，建议将完整内容放置于独立的 `.md` 文档中，并在注解处保留内部引用链接。

**问：我能否在 Olive Resource Atlas 中收录非公开或需要登录权限的内部系统链接？**

答：原则上不推荐收录需要特殊权限或内网访问的链接，因为这会导致外部贡献者无法验证链接的有效性，并可能引发安全风险。若确有必要收录内部系统入口，请在注解中明确标注“内部使用”以及所需的访问权限等级，并确保该链接不会暴露敏感信息。对于企业内部部署场景，建议将此类条目放入单独的私有分支中管理，不合并至公共主干。
