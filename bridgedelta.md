# Olive Index

Olive Index 是一个面向开发者、技术研究人员与内容策展人的轻量化外链资源汇总与结构化导航工具。该项目定位为“技术资源的静态索引层”，不对资源内容本身进行托管或二次分发，而是通过标准化的 Markdown 索引文件，将分散在互联网各处的优质技术文档、代码仓库、社区讨论与规范文本进行集中归类与语义标注。Olive Index 适用于需要维护个人技术书签库、构建团队知识导航门户，或为开源项目提供附属资源索引的中小型场景。其核心设计原则为：零数据库依赖、纯静态部署、条目级版本可追溯、以及人类可读的目录结构。

## 功能概览

- 语义化资源分类索引：基于文件命名约定与目录层级，对资源链接进行主题标记与分组，支持多维度筛选与快速定位。

- 多协议资源引用支持：统一处理裸域名、HTTP 与 HTTPS 协议链接，保留原始 URL 格式，避免自动补全或协议改写带来的访问异常。

- 可嵌入的轻量级导航页：提供纯 HTML/CSS 的浏览界面，适合部署为 GitHub Pages、Nginx 静态目录或 CDN 分发站点。

- 条目级变更追踪：每个索引条目以独立 Markdown 文件存储，配合 Git 提交历史，可清晰追溯资源的添加、删除与描述更新。

- 自动校验与链接健康检查：集成基于 GitHub Actions 的定时工作流，扫描索引文件中所有外链，标记失效或状态异常的 URL 并生成报告。

- 批量导入与模板生成：支持从 CSV 或 TSV 文件批量生成索引条目模板，降低大规模资源录入的重复操作成本。

- 标签系统与全文检索：为每个资源条目附加自由标签，并借助静态站点生成器实现简单的关键词全文检索功能。

## 应用场景

- 个人技术书签库的版本化管理：开发者可将日常阅读的技术博客、API 文档、规范草案等链接以索引条目形式纳入 Olive Index，通过 Git 在设备间同步，并利用提交信息记录收藏原因或阅读状态。

- 团队内部知识导航门户：技术团队可使用 Olive Index 搭建私有部署的导航页，集中存放常用的内部系统地址、运维手册、设计文档与项目仓库链接，避免浏览器书签分散与过期问题。

- 开源项目的附属资源索引：开源项目维护者可将项目相关的扩展工具、示例代码、社区翻译、视频教程等外部资源整理为 Olive Index 索引，放置于项目 docs 目录下，方便贡献者与用户按需查阅。

- 技术写作与研究报告的参考资料库：技术博主或研究人员可在撰写长文或报告时，使用 Olive Index 集中管理所有引用链接，并在文章末尾生成可维护的资源清单，减少死链风险。

- 离线环境下的资源目录分发：对于内部网络受限的环境，可定期导出 Olive Index 的静态索引页面，与资源本体一同打包，实现离线场景下的导航与检索。

## 快速开始

以下命令演示了从克隆仓库到启动本地预览服务的完整流程。

```bash
# 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（基于 Node.js 环境，用于本地预览与校验）
npm install

# 启动本地开发服务器，默认监听端口 3000
npm run dev
```

执行上述命令后，访问控制台输出的本地地址即可查看索引导航界面。如需构建生产版本，请执行 `npm run build`，产物将输出至 `dist` 目录。

## 贡献指南

1. 查阅贡献者行为准则与项目路线图，确认您的提议与项目方向一致。建议先在议题列表中搜索是否已有类似提议，避免重复工作。

2. 从仓库派生副本至您的个人账号，并在本地创建功能分支。分支命名建议遵循 `feat/描述` 或 `fix/描述` 的格式。

3. 按照 `templates/entry-template.md` 的格式新增或修改索引条目，并确保所有外链 URL 保持原始格式，不添加多余前缀或尾部斜杠。新增条目后，请在本地执行 `npm run validate` 进行格式与链接可达性检查。

4. 提交变更时，请编写清晰且符合常规提交规范的提交信息，例如 `feat: add resource index for WebAssembly proposals`。提交前需确保所有自动化检查通过，包括类型检查、代码风格与链接校验。

5. 向主仓库的 `main` 分支发起拉取请求，并在描述中关联相关议题编号（如有）。项目维护者将在三个工作日内进行评审，必要时会提出修改意见。合并后，您的变更将自动触发生产环境重新部署。

## 常见问题

问：索引文件中的 URL 是否需要进行百分号编码或规范化处理？

答：不需要。Olive Index 强制要求索引条目中的 URL 必须与用户提供的原始字符串完全一致，包括协议前缀、域名大小写以及路径中的特殊字符。系统不会进行任何自动补全或转义，以确保资源链接的可预测性与直接可访问性。但请注意，部分老旧浏览器可能无法识别未编码的空格，建议在源文件中避免此类字符。

问：链接校验工作流报告了某个 URL 为“失效”，但我通过浏览器可以正常访问，是什么原因？

答：这种情况通常由以下原因导致：第一，校验工具的请求头与浏览器不同，某些网站会返回 403 或 429 状态码以阻止非浏览器流量；第二，目标服务器对请求频率有限制，工作流可能在短时间内触发了速率限制；第三，部分站点要求特定的 Cookie 或 Referer 头。如果确认链接本身有效，您可以在索引条目中添加 `ignore-check: true` 的元信息来跳过该链接的自动校验，并附上简短说明。

问：是否支持嵌套分类或层级标签？

答：当前版本采用扁平标签体系与单层目录分类的组合方式。您可以通过在文件名中使用连字符（例如 `networking-http3.md`）来模拟逻辑层级，但系统本身不会解析层级关系。嵌套分类已在后续版本的规划中，届时会提供向后兼容的迁移方案。
