# Olive Field Reference

Olive Field Reference 是一个面向技术文档归档与外部知识库索引的开源元数据仓库。本项目不对任何外部资源进行内容镜像，而是通过结构化 Markdown 文件维护一组经过筛选的技术参考链接、领域术语注解和概念映射关系，帮助技术团队、文档编写者和研究者快速定位特定领域的高价值原始资料。项目定位为“轻量级外链与术语指南”，适用于需要频繁查阅外部技术规范、协议说明、开源项目文档或行业标准原文的场景。

本项目不提供搜索引擎或动态抓取服务，所有资源均以静态列表形式存储于仓库目录中，用户可通过本地阅读或版本控制历史追溯链接的增删与变更。项目目标用户包括：需要维护私有技术书签库的开发者、撰写技术博客或论文时需引用权威外源的作者、以及希望减少重复检索耗时的运维与架构师。

## 功能概览

- **结构化外链索引**：每个 Markdown 文件对应一个特定主题或组件，内部以无序列表或表格形式收录相关外部链接，所有链接均保留原始协议与域名格式。

- **分领域术语注解**：针对每个外链资源，附加简短的术语解释或使用背景说明，帮助读者判断链接是否与其当前问题相关。

- **版本化变更记录**：通过 Git 提交历史追踪每个链接文件的增删改操作，支持回溯任意时间点的外链集合状态。

- **零依赖静态查阅**：无需安装任何数据库或后端服务，仅需 Markdown 阅读器即可浏览全部内容，兼容本地编辑器与 GitHub 网页渲染。

- **外链有效性自检辅助**：提供可选的外部脚本（非必需），用于定期检查列表中各 URL 的可访问状态，并将结果输出为日志文件，辅助维护者清理失效链接。

- **多主题分类覆盖**：当前仓库已包含若干预置主题文件，涵盖基础架构、数据协议、开发工具链、运维监控等常见技术域，用户亦可自行新增文件扩展分类。

## 应用场景

- **技术选型调研**：当团队需要评估多个开源项目或商业产品时，可直接查阅本项目对应主题文件中的官方文档链接、性能对比报告原文地址以及社区讨论汇总，避免从零开始搜索。

- **文档编写引用规范**：技术文档编写者可在撰写系统设计说明书或 API 文档时，使用本项目的链接列表作为标准引用源，确保所有外链均经过初步筛选且格式统一。

- **新人培训路径指引**：将本项目作为新入职工程师的初始阅读材料之一，帮助其快速了解团队常用技术栈的官方文档位置、核心规范原文以及推荐的学习参考资源。

- **离线知识库搭建基础**：用户可基于本项目提供的链接列表，自行使用离线下载工具批量获取外部页面内容，构建内部隔离网络环境下的备用知识库。

- **学术研究参考文献整理**：研究人员可将本项目作为参考文献来源的临时存储区，在撰写论文过程中随时添加或分类管理各类技术标准、实验数据来源或开源代码仓库链接。

## 快速开始

以下命令演示如何在本地克隆本项目并查看已有资源列表。

```bash
# 克隆仓库到本地
git clone https://github.com/zerxonhy/olive.git
cd olive

# 查看主索引文件（若存在）或直接浏览各主题文件
ls -la *.md

# 使用任意文本编辑器打开一个主题文件，例如查看 fieldbloom 相关链接
cat fieldbloom.md

# （可选）若需运行辅助检查脚本，请确保已安装 curl 或 wget，并赋予执行权限
# chmod +x scripts/check-urls.sh
# ./scripts/check-urls.sh
```

## 安装要求

本项目以静态 Markdown 文件为主，无强制运行时依赖。若用户希望使用辅助检测脚本，需满足以下可选条件：

| 依赖项 | 必需性 | 说明 |
|--------|--------|------|
| Git | 必需 | 用于克隆仓库及查看提交历史 |
| Markdown 阅读器 | 必需 | 任意支持 Markdown 渲染的编辑器或浏览器（GitHub 原生支持） |
| Bash 4.0+ | 可选 | 仅当需要运行附带的外链检测脚本时需 Bash 环境 |
| curl 或 wget | 可选 | 检测脚本使用此类工具发送 HTTP 请求验证链接状态 |
| grep / awk | 可选 | 检测脚本中用于解析 URL 列表的文本处理工具 |
| 网络连接 | 可选 | 仅在访问外部链接或在线浏览 GitHub 页面时需要 |

## 贡献指南

1. 复刻本项目至个人账户，并在本地新建分支进行修改。建议分支命名格式为 `topic/your-topic-name` 或 `fix/description`。

2. 在对应主题的 Markdown 文件中新增或修改外部链接条目。每个链接必须单独占一行，并附带简短的上下文说明（如“官方文档”、“性能基准报告”、“社区讨论帖”）。禁止删除或修改他人已标注为“锁定”的链接，除非同时提交说明理由。

3. 提交变更前，请运行本地检测脚本（若已配置）确认新增链接可访问，并确保 Markdown 表格或列表格式正确无误。提交信息应使用清晰的行为动词，例如“add: 新增 Kubernetes 1.28 调度器说明链接”、“update: 更新 TLS 1.3 规范原文地址”。

4. 向主仓库发起 Pull Request，描述变更目的、影响范围以及是否涉及破坏性改动（如删除原有链接）。PR 描述中需勾选对应的自检项（如“已运行检测脚本”、“未重复添加已有链接”）。

5. 等待维护者审核。审核期间可能会要求补充说明或调整链接分类。合并后，变更即进入主分支并随下次 Git 标签发布生效。

## 常见问题

**问：本项目是否存储外部资源的副本或缓存内容？**

答：不存储。本项目仅维护指向原始外部资源的 URL 链接，所有内容版权及可用性均归原始发布者所有。用户访问外部链接时需遵守对应网站的使用条款。

**问：如果发现某个链接失效，应该如何处理？**

答：用户可通过 GitHub Issues 提交失效链接报告，标题格式建议为“Broken link: [文件名] - [链接片段]”。维护者会定期核实并移除或替换失效链接。用户亦可自行按贡献指南提交 Pull Request 修复，但需在提交信息中注明链接失效日期及替代链接来源。

**问：能否在项目内新增自定义主题文件？**

答：可以。用户可在仓库根目录下新建 `*.md` 文件，并参考现有文件的结构编写。建议在文件顶部使用一级标题说明主题范围，内部使用二级标题划分小节，链接列表统一置于各小节末尾。新增文件后需在 `goldenatlas.md` 的全局映射表中补充对应条目，以便其他用户检索。
