# Olive Resource Aggregator

Olive Resource Aggregator 是一个面向开发者和技术研究人员的轻量级外链资源汇总与导航工具。本项目旨在解决技术文档、代码示例、设计规范等分散于多个仓库和站点、难以统一检索与引用的问题，通过结构化的 Markdown 索引文件，将外部优质技术资源以可追溯、可版本化的方式归集于单一项目内，便于团队或个人建立私有技术知识缓存。

项目定位为纯静态资源索引层，不依赖数据库、后端服务或第三方索引引擎，仅通过 GitHub 仓库内的 Markdown 文件维护外部链接清单。目标用户包括技术文档撰写者、开源项目维护者、技术调研团队以及需要频繁查阅外部参考实现的开发人员。通过本项目的索引机制，用户可快速定位到经过筛选的高质量技术文章、代码片段、配置模板与架构方案，显著降低信息检索成本。

## 功能概览

- **按主题分类的资源索引**：资源文件按技术领域或功能模块划分，每个 Markdown 文件对应一类主题，便于按需查阅。

- **外部链接持久化存档**：每个资源条目记录原始 URL、标题、摘要及收录时间，形成可追溯的引用快照，防止源链接失效后信息丢失。

- **全文检索友好的前置元数据**：每个索引文件包含 YAML Front Matter，支持通过静态站点生成器或 IDE 插件进行快速全文检索。

- **版本化更新历史**：资源增删改通过 Git 提交记录完整保留，支持回溯任意时间点的资源列表状态，满足审计与变更管理需求。

- **Markdown 原生渲染**：所有索引文件使用标准 Markdown 语法编写，可在 GitHub、GitLab 或任何支持 Markdown 的平台上直接预览，无需额外工具。

- **资源状态标记**：每条资源可标注“有效”、“失效待确认”、“已迁移”等状态，帮助用户判断链接可用性。

- **自定义标签系统**：允许为每条资源添加多个自定义标签，实现跨分类的交叉检索，例如按编程语言、框架版本或应用场景筛选。

- **定期健康检查脚本**：项目附带轻量级 Shell 脚本，可定时检测索引中所有外链的 HTTP 状态码，生成可用性报告。

## 应用场景

- **技术团队内部知识库构建**：团队可将日常调研、项目复盘、技术选型过程中发现的优质外部资源统一收录至本项目索引中，形成共享的知识基线，新成员入职时可快速了解团队所依赖的外部技术生态。

- **开源项目参考文档聚合**：开源项目维护者可将项目依赖的规范文档、协议标准、上游依赖说明等外部链接集中管理，减少在各处散落链接带来的维护负担，同时为贡献者提供清晰的上游信息入口。

- **技术调研与竞品分析**：在进行技术选型或竞品分析时，分析师可将收集到的竞品技术栈、官方文档、社区讨论帖、性能测试报告等链接按主题归档，便于后续横向对比与结论回溯。

- **个人学习路径管理**：开发者可将学习过程中遇到的教程、API 参考、最佳实践示例等资源按学习阶段或技能领域分类，配合状态标记功能记录学习进度，形成可复用的个人学习资源库。

## 快速开始

以下命令演示如何从 GitHub 克隆本项目、安装基础依赖并运行内置的资源健康检查脚本。

```bash
# 克隆项目至本地
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（需具备 Node.js 和 npm 环境）
npm install

# 运行资源链接健康检查（检查所有索引文件中的外链可用性）
npm run check:links

# 生成静态资源索引报告（输出至 ./reports 目录）
npm run build:index
```

## 贡献指南

1. 复刻本仓库至个人 GitHub 账户，并在本地克隆复刻后的仓库。创建新的功能分支，分支命名建议遵循 `feat/resource-update-<日期>` 或 `fix/link-status-<资源名>` 的格式。

2. 在对应的主题 Markdown 文件中新增或修改资源条目。每一条资源必须包含原始 URL、标题、简短摘要以及收录日期。若需新增主题分类，可在项目根目录新建 `*.md` 文件，并同步更新本 README 中的项目结构说明与文档导航表格。

3. 提交变更前，请运行 `npm run check:links` 确保所有新增或修改的外链返回有效的 HTTP 成功状态码（2xx 或 3xx）。对于临时不可用的链接，请标记状态为“失效待确认”并注明最后检测时间。

4. 推送分支至远程复刻仓库，随后通过 GitHub 界面发起 Pull Request 至本仓库的 `main` 分支。PR 描述中请简要说明本次资源更新的目的、涉及的文件列表以及链接状态检查结果摘要。

5. 项目维护者将在 3 个工作日内审查 PR。审查通过后，PR 将被合并，合并后 GitHub Actions 将自动执行全量链接检查并更新报告。

## 常见问题

**Q: 索引文件中的链接失效了怎么办？**

A: 项目内置的链接健康检查脚本会每日定时运行（通过 GitHub Actions）。当检测到失效链接时，系统会自动在仓库的 Issues 中创建提醒工单，并标记对应的资源条目状态为“失效待确认”。贡献者可认领工单，访问原始链接确认是否永久失效或仅临时波动。若永久失效，应从索引中移除该条目并提交 PR；若为临时波动，可将状态更新为“有效”并重置检测时间。

**Q: 能否在本项目中添加非技术类的外链资源？**

A: 本项目定位为技术资源汇总站，原则上仅收录与软件开发、系统运维、架构设计、信息安全等计算机领域直接相关的技术文档、规范标准、代码示例和工程实践类链接。对于非技术类资源（如新闻资讯、社交媒体讨论、商业推广页等），将不予收录。若不确定某链接是否符合范围，可先提交 PR 并附上说明，由维护者评估决定。

**Q: 如何批量导入已有的书签或收藏夹中的链接？**

A: 目前项目未提供自动化导入工具，但您可借助浏览器书签导出功能（通常为 HTML 格式），配合简单的文本处理脚本（如使用 `grep` 和 `sed` 提取 URL 和标题）生成符合 `resource-template.md` 格式的条目。建议每次导入不超过 20 条，以便手动校验每条链接的质量与分类准确性。若需导入大量链接，请在 PR 中说明导入背景和筛选标准。
