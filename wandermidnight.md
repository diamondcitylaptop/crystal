# Horizon Index

Horizon Index 是一个面向技术研究、系统运维与知识工程领域的轻量化外链聚合与结构化导航工具。该项目不提供内容托管，而是以可版本化的 Markdown 清单形式，对分散在代码托管平台、技术社区与个人站点的优质深度内容进行编目、分类与注释。其目标用户包括需要维护团队知识图谱的架构师、进行大规模文献调研的研究人员，以及希望建立个人技术阅读队列的开发者。通过将资源索引与项目代码仓库解耦，Horizon Index 使外链资源能够像代码一样接受 PR 审查、议题讨论与版本回溯，从而解决传统书签工具缺乏协作能力、链接失效后无法追溯上下文、以及难以与工程文档融合的长期痛点。

## 功能概览

- **扁平化资源编目**：所有收录链接以纯文本形式存储于项目根目录下的指定 Markdown 文件中，每条记录包含标题、原始 URL、采集日期与简短摘要，支持全局搜索与批量导入。

- **自动化元数据提取**：项目集成了基于 GitHub API 与网页元标签解析的轻量级脚本，在资源入库时自动抓取页面标题、描述与最后修改时间，减少人工录入成本。

- **版本化变更追踪**：依托 Git 原生能力，每次增删或修改资源链接都会生成可追溯的提交记录。维护者可通过提交信息快速定位某条链接的引入原因或弃用决策。

- **标签化分类系统**：每条资源可赋予一到多个分类标签（如 `#observability`、`#distributed-system`、`#rust`），项目维护了一套标签别名表，确保分类体系的一致性与可扩展性。

- **链接健康检查**：项目提供了独立的 CI 工作流，定期（每周一次）对所有收录链接进行 HTTP 状态探测，并在检测到 4xx/5xx 错误或 TLS 证书异常时自动创建议题，提醒维护者介入处理。

- **外部引用关联图**：通过解析资源正文中引用的其他外链，项目能够生成简单的关联图谱（基于 Graphviz），帮助使用者发现主题集群与核心枢纽节点。

## 应用场景

- **团队技术雷达维护**：技术委员会可使用 Horizon Index 维护每季度的技术雷达清单，将备选工具、框架或论文的链接统一收录，并利用议题功能开展投票与讨论，最终将采纳项标记为推荐。

- **个人研究文献队列管理**：研究员在阅读论文或技术博客时，可将待读或已读的深度内容通过 PR 形式提交至自己的私有分支，利用项目的标签系统区分「待精读」、「已归档」与「待验证」，形成可检索的个人知识边栏。

- **开源项目外部依赖文档化**：开源项目维护者可将项目依赖的底层规范文档、参考实现仓库或性能测试基准站点纳入 Horizon Index，作为项目 README 或 Wiki 的补充外链索引，降低新贡献者的学习曲线。

- **离线文档构建素材收集**：文档工程师在构建离线帮助中心或技术电子书时，可使用 Horizon Index 收集待引用的外部规范与标准正文链接，并通过项目的元数据导出功能生成 CSV 清单，方便后期进行版权核查与链接校正。

## 快速开始

以下指令适用于 Linux / macOS / Windows WSL 环境，假设系统已安装 Git 与 Node.js 18.x 或以上版本。

```bash
# 克隆项目仓库至本地
git clone https://github.com/your-organization/horizon-index.git
cd horizon-index

# 安装项目依赖（包括元数据抓取工具与健康检查脚本）
npm install

# 运行本地资源索引构建脚本，生成初始 catalog.json 与静态站点预览
npm run build

# 启动本地开发服务器，默认监听 8080 端口，访问 http://localhost:8080 查看索引面板
npm start
```

## 贡献指南

我们欢迎并鼓励外部贡献者以 Pull Request 形式提交新的资源链接或改进现有元数据。为保证索引质量，请遵循以下步骤：

1. **查阅现有条目与标签规范**：在 `/catalog/entries/` 目录下搜索是否已有相似主题资源，避免重复收录；同时阅读 `/docs/taxonomy.md` 确认新链接的标签符合当前分类体系，若需新增标签，请先在 `/catalog/tags/` 下提交别名定义文件。

2. **创建新条目文件**：在 `/catalog/entries/` 下以 `[slug].md` 格式命名文件（slug 为 URL 路径的摘要哈希），并严格按照模板填写 `# Title`、`- URL:`、`- Date:`、`- Tags:` 与 `- Summary:` 字段。

3. **本地构建验证**：运行 `npm run build` 确保新条目能够被解析器正确读取，且没有破坏静态站点的生成流程。若构建失败，请根据终端错误提示修正条目格式。

4. **提交 Pull Request**：将变更推送到你的个人分支，并在 PR 描述中简要说明该资源的推荐理由、所属领域以及是否替代或更新了已有条目。PR 至少需要一名项目维护者的 Approve 方可合并。

5. **响应 CI 检查**：合并前，GitHub Actions 会自动运行格式检查与链接可达性预检。若检查失败，请根据日志修正后再更新 PR。合并后，新的条目将在下一次元数据刷新工作流中自动补全标题与最后修改时间。

## 常见问题

**Q: 如果收录的链接失效了，项目会自动处理吗？**

A: 项目内置了每周执行的 CI 健康检查任务。当检测到某条链接返回 HTTP 状态码 404、500 或发生 TLS 握手失败时，系统会在当前仓库自动创建一个议题，标题包含失效链接的条目 ID，正文附带上一次成功访问的记录时间与当前错误详情。维护者收到议题通知后，可以手动验证链接是否临时性故障，若确认永久失效，则通过提交移除该条目或替换为更新后的 URL。

**Q: 我能否将本项目只作为个人书签使用，而不公开我的索引内容？**

A: 完全可以。Horizon Index 不强制要求公开仓库。你可以将项目克隆为私有仓库，或在本地使用，不推送到任何远程。所有脚本均支持离线运行模式（仅依赖本地缓存），元数据抓取功能在没有网络代理的情况下也能正常处理已有条目。如果你希望保留协作能力但不公开内容，可以考虑自托管 Git 服务（如 Gitea）并设置私有可见性。

**Q: 项目的静态站点预览与最终生成的产物有什么区别？**

A: 本地执行 `npm start` 启动的开发服务器会生成一份实时预览的索引面板，主要用于编辑与调试目的，包含所有未提交的本地变更。而通过 `npm run build` 生成的产物位于 `/dist` 目录下，是完整的静态 HTML 文件集合，可用于部署到任何 Web 服务器或对象存储上。生产环境部署建议使用构建产物，以获得更优的缓存策略与压缩资源。
