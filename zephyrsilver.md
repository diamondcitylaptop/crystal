# Olive Resource Atlas

Olive Resource Atlas 是一个面向开发者与研究工作者的技术资源外链聚合与导航系统。该项目不存储任何实际数据文件，而是以结构化索引方式收集、分类并维护高质量的外部技术文档、项目参考与知识库链接。项目定位为“技术资源的精选入口”，适用于需要快速定位特定领域参考材料、对比不同实现方案或追踪技术演进脉络的用户群体。Olive Resource Atlas 通过严格的链接审核机制与版本化索引记录，确保资源可用性与引用可追溯性，解决传统书签管理工具缺乏上下文、缺乏分类维度以及协作困难等问题。

## 功能概览

- 按领域分类的资源索引 系统将收录的外部链接按技术领域、应用场景与成熟度进行多维度分类，便于用户按需筛选。

- 链接状态健康检查 定期对已收录的 URL 执行可用性探测，自动标记失效或重定向的链接，并在索引状态中予以标注。

- 版本化引用记录 每次资源增删或属性修改均生成变更日志，支持按时间点回溯索引状态，满足研究项目对引用稳定性的要求。

- 标签与全文检索 每个资源条目允许附加多个自定义标签，同时支持对资源标题、描述、分类标签进行关键词全文检索。

- 外部元数据快照 对每个收录的链接，系统缓存其页面标题、描述摘要与关键正文片段，用于在索引页面中生成上下文预览。

- 协作审核工作流 注册用户可提交新资源推荐，经由维护者审核通过后正式纳入索引，同时支持对已有资源发起属性修正请求。

- 导出与集成接口 支持将索引数据导出为 JSON、CSV 以及 Markdown 表格格式，并提供 RESTful API 供第三方工具集成。

## 应用场景

- 技术选型调研 技术负责人或架构师在评估不同中间件、数据库或前端框架时，可通过 Olive Resource Atlas 快速获取相关领域的外部对比文章、性能测试报告与社区讨论链接，统一汇集在项目索引中供团队参考。

- 研究文献参考管理 高校研究人员或实验室成员在撰写论文或技术报告时，可将引用的外部技术规范、开源项目仓库或数据手册通过本系统统一收录，并利用版本化索引记录生成引用清单，避免手动整理时的遗漏或链接丢失。

- 新人入门学习路径规划 团队为新入职开发者制定学习计划时，可在 Olive Resource Atlas 内创建专属分类，将内部培训材料与外部优质教程、视频课程、官方文档链接一并整合，形成可共享的学习导航页面。

- 技术社区内容聚合 社区运营者或开源项目维护者可将社区内产生的高质量讨论帖、博客文章、案例研究链接集中收录，减轻社区成员自行检索的负担，同时通过标签检索快速定位特定主题。

## 快速开始

以下命令序列演示如何从 GitHub 克隆项目仓库、安装依赖并启动本地开发服务。

```bash
git clone https://github.com/zerxonhy/olive.git
cd olive/resource-atlas
npm install
npm run dev
```

执行完成后，访问控制台输出的本地地址即可浏览索引首页。生产环境部署请参考 `docs/deployment.md` 中的说明。

## 贡献指南

1. 复刻项目仓库至个人账号，并在本地基于 `main` 分支创建新的特性分支，分支命名建议采用 `feat/`、`fix/` 或 `docs/` 前缀加简短描述。

2. 完成代码或文档修改后，请确保所有现有测试用例通过，并为新增功能补充相应的单元测试或集成测试，测试文件放置于 `tests/` 目录下对应子目录。

3. 提交变更时遵循项目规定的提交信息格式，即 `<type>(<scope>): <subject>`，其中 type 可选 `feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`。

4. 推送分支至个人复刻仓库，然后通过 GitHub 界面发起 Pull Request 请求，目标分支为上游仓库的 `main` 分支。请求描述中需清晰说明变更内容、动机以及是否涉及破坏性改动。

5. 项目维护者将在收到请求后的 5 个工作日内进行审核，可能要求进一步修改或补充信息。审核通过后由维护者执行合并操作。

## 常见问题

**问：我提交的资源推荐被拒绝，可能的原因有哪些？**

答：常见拒绝原因包括：链接内容与现有分类领域明显不符；目标站点存在大量广告或恶意弹窗；链接指向的页面内容过于单薄或已明显过时；提交信息中缺少必要的标题或描述说明。建议在重新提交前仔细阅读 `docs/maintainer/review-criteria.md` 中的详细审核标准。

**问：是否可以完全脱离 Redis 运行本系统？**

答：可以。系统设计上对 Redis 采用可选依赖策略。当环境变量中未配置 Redis 连接字符串时，系统会自动降级为内存缓存与 SQLite 队列实现，但此时多进程部署场景下的缓存一致性与定时任务的可靠性将有所降低，单机开发或小规模内部使用不受影响。

**问：如何批量导入已有的书签集合或浏览器收藏夹？**

答：系统提供了 `scripts/import-from-html.js` 工具，支持将 Chrome 或 Firefox 导出的 HTML 书签文件转换为系统内部索引格式。具体使用方法请参考 `docs/maintainer/batch-import.md` 文档，该文档同时说明了字段映射规则与标签生成策略。
