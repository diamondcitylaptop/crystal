# Olive Link Atlas

Olive Link Atlas 是一个面向开发者、技术研究人员与开源贡献者的结构化外链与资源汇总工具。该项目不提供原生成品内容，而是通过严格的 Markdown 文件索引机制，将分散于 GitHub 仓库中的技术笔记、配置方案、架构决策记录与实验性文档进行集中归类与导航。Olive Link Atlas 定位为“技术资源的资源”，帮助用户在海量代码与文档碎片中快速定位高价值的外部参考链接，适用于个人知识管理、团队技术栈审计、开源项目依赖溯源等场景。

目标用户包括：需要维护大量外部参考链接的技术文档工程师、进行竞品或组件调研的架构师、希望从他人公开笔记中获取灵感的开源初学者，以及需要快速重建开发环境或部署流程的运维人员。Olive Link Atlas 不替代搜索引擎或包管理器，而是作为人工筛选与语义标注后的链接索引层，提升信息检索的确定性与可追溯性。

## 功能概览

- **多级目录索引**：支持按主题、时间、版本或项目阶段对链接进行树状归类，每个链接条目均附带来源文件路径与最后更新时间戳。

- **原始内容透传**：所有外链均以裸 URL 或完整协议 URL 原样呈现，不做跳转、短链压缩或内容代理，确保指向的原始资源可被直接访问。

- **上下文注释继承**：每个链接条目继承自源 Markdown 文件中的前后文注释，包括作者、用途说明、已知限制或替代方案，方便后续审核。

- **版本差异标记**：若同一资源存在多个版本或镜像地址，通过文件命名后缀（如 cedarwillow 与 coralmaple）进行区分，并附带差异摘要。

- **快速失败检测**：项目内置简单的 HTTP 状态码检查脚本（基于 curl），可定期标记失效链接，辅助维护者进行清理或更新。

- **标签过滤系统**：每条记录支持自由标签（如 #deployment, #api, #security），允许用户通过 grep 或 IDE 搜索进行快速筛选。

- **离线阅读支持**：所有索引文件均为纯 Markdown，克隆后无需网络即可浏览链接结构，适合内网或离线环境使用。

## 应用场景

- **技术选型调研**：当团队需要评估多个开源数据库或消息队列时，可通过 Olive Link Atlas 中预收集的对比文章、性能测试报告和官方文档链接，快速建立信息全景，减少重复搜索时间。

- **环境初始化与灾难恢复**：运维人员可在新服务器或容器中克隆该项目，按索引顺序依次访问配置脚本、镜像仓库地址和密钥管理指南，确保每次部署步骤一致且可追溯。

- **开源项目依赖追踪**：开发者维护一个依赖较多外部库的项目时，可将每个依赖的官网、Issue 列表、迁移文档和社区讨论贴统一收录，当依赖发生重大更新时，能够迅速评估影响范围。

- **技术文章写作素材管理**：技术博主或文档作者在撰写系列教程时，利用 Olive Link Atlas 的目录结构组织参考来源，既保证引用的准确性，也便于读者一键跳转原文。

- **团队交接与入职培训**：新成员通过浏览该项目的链接树，可以系统性地了解团队常用的工具链、规范文档、监控面板和内网服务地址，缩短熟悉周期。

## 快速开始

以下步骤帮助您在本地环境快速部署并运行 Olive Link Atlas 的辅助检查工具。

```bash
# 克隆仓库（包含所有索引文件）
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（仅需 curl 和标准 GNU 工具）
# 本步骤为环境检查，若已安装可跳过
sudo apt-get update && sudo apt-get install -y curl coreutils grep sed

# 运行链接状态检查脚本（示例）
# 该脚本会读取 ./links.txt 或扫描所有 .md 文件中的外链
bash scripts/check-links.sh
```

## 贡献指南

1.  **克隆与分支**：Fork 本仓库，并在您的副本中创建一个新的功能分支，分支名应反映本次变更内容，例如 `add-cedar-notes` 或 `fix-broken-link-2026`。

2.  **遵循文件命名规则**：新增笔记或索引文件时，统一使用小写字母与连字符，确保名称具有描述性。若更新某一现有资源文件，需在文件头部注释中更新 `Last-Modified` 日期。

3.  **更新链接列表**：若增加或移除外部资源，必须同步修改 `links.txt` 以及 `indexes/` 下的相关索引文件，确保所有列表保持一致。每个 URL 必须保持原始形式，不添加或修改协议与域名。

4.  **本地验证**：在提交前，运行 `scripts/format-validator.sh` 检查文档语法，并执行 `scripts/check-links.sh` 确保所有新增或变更链接至少返回 HTTP 2xx 或 3xx 状态码。若链接不可达，需在笔记中注明已知问题。

5.  **发起拉取请求**：将您的分支推送到 GitHub，并发起 Pull Request 至主仓库的 `main` 分支。请在描述中清晰列出本次变更的文件列表、变更目的以及任何例外情况（如永久失效但需保留的链接）。

## 常见问题

**问：Olive Link Atlas 是否存储或缓存外部资源的内容副本？**

答：否。本项目仅存储指向外部资源的 URL 及相应的文本注释，不缓存、不代理、不备份任何外部文件内容。所有资源均实时从原始地址获取，用户需自行承担访问外部链接的合规性与安全性责任。

**问：某个链接已经失效，我应该如何处理？**

答：若您在使用过程中发现链接返回 404 或连接超时，请先通过搜索引擎确认该资源是否已迁移至新地址。若找到新地址，可按贡献指南提交 PR 更新 URL 并保留旧链接至归档目录。若资源已彻底移除，请在对应笔记中标记 `[DEPRECATED]` 并注明最后检查时间，同时从主索引中移出。

**问：如何批量导出所有链接用于其他工具？**

答：您可以直接使用项目根目录下的 `links.txt` 文件，该文件以纯文本格式每行一个 URL 列出所有活跃链接。此外，您可以运行 `scripts/generate-index.sh --format=csv` 生成包含标签和描述的 CSV 格式导出文件，便于导入电子表格或自定义脚本。
