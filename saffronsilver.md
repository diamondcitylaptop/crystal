# Olive Resource Navigator

Olive Resource Navigator 是一个面向开发者与研究人员的技术资源外链汇总与导航系统，专注于将分散在多个仓库、文档站点与知识库中的高价值引用链接进行结构化归集与版本化呈现。该项目本身不存储任何实际内容文件，而是通过 Markdown 索引文件对外部资源进行语义化标注、分类组织和变更追踪，适用于需要长期维护外部参考链接列表的技术文档库、知识管理仓库或项目孵化期的资源看板。

该项目定位为轻量级、纯静态、基于 Git 版本控制的资源链接枢纽站，主要服务于技术作者、开源维护者、文档工程师以及需要频繁引用外部规范、教程、工具或案例代码的研发团队。Olive Resource Navigator 通过统一的索引文件格式与目录规范，将原本散落在个人收藏夹、临时笔记或聊天记录中的链接转化为可审查、可追溯、可协作的公开资源清单，从而降低团队内部信息漂移与链接失效风险。

## 功能概览

- **链接索引化存储** 每个外部资源链接均以独立条目形式记录在指定的 Markdown 索引文件中，包含原始 URL、资源类型、简短描述与采集时间，便于后续检索与审计。

- **多维度分类标签** 支持为每个链接赋予多个分类标签，例如 "tutorial"、"api-reference"、"code-example"、"specification"、"community-tool" 等，允许按主题或用途快速筛选。

- **版本变更追踪** 由于全部索引文件以纯文本形式托管在 Git 仓库中，每一次链接的新增、删除或更新均可通过提交记录与差异对比进行追溯，支持回滚与变更审查。

- **资源状态标记** 提供可扩展的状态字段，用于标记链接的可用性、维护优先级或审核状态，例如 "stable"、"deprecated"、"under-review" 或 "broken"，帮助团队识别需要关注的条目。

- **批量导入与校验** 内置简单的脚本工具，支持从 CSV 或 JSON 格式的旧有链接清单批量导入条目，并在导入过程中对 URL 格式与重复性进行基础校验，减少人工录入错误。

- **静态站点生成适配** 索引文件设计为与常见静态站点生成器（如 Hugo、Jekyll、VuePress）兼容，可快速将链接列表渲染为 HTML 导航页面，便于内部发布或对外展示。

- **协作审查工作流** 通过 Pull Request 流程管理链接的新增与修改，强制要求填写变更说明与资源验证结果，确保每一枚外链均经过必要的人工检查。

## 应用场景

**技术文档库的外部参考管理**  
当团队维护大型技术文档或教程集合时，常常需要引用外部规范、SDK 文档或社区最佳实践。Olive Resource Navigator 可用于集中管理这些引用链接，并在文档版本升级时统一检查链接有效性，避免文档中出现过时或失效的参考地址。

**开源项目孵化期的资源看板**  
在开源项目早期阶段，核心开发者需要持续跟踪相关的技术文章、示例代码、竞品资料或社区讨论。该导航系统可充当项目内部的资源看板，帮助新加入的贡献者快速了解项目所涉及的外部知识体系，降低入门门槛。

**技术培训与学习路径组织**  
技术培训负责人可以将课程中涉及的阅读材料、实验环境地址、视频教程链接等统一收录为索引条目，并按学习阶段或模块进行标签划分。学员可通过导航文件获得清晰的学习路径，无需在多个文档间跳转查找链接。

**多仓库联合文档的链接中枢**  
对于采用微服务架构或多仓库协作的项目，各个子仓库的文档可能相互引用。Olive Resource Navigator 可作为独立的链接中枢仓库，集中记录所有跨仓库引用地址，当某个子仓库地址变更时，仅需更新中枢文件即可，无需逐个修改各个仓库的文档。

## 快速开始

以下命令演示如何获取项目代码、安装基础依赖并启动本地预览服务。

```bash
# 克隆仓库到本地
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装 Python 3.9+ 与 pip 依赖（用于链接校验与索引生成工具）
pip install -r requirements.txt

# 运行本地静态站点预览（基于内置的简易 HTTP 服务器）
python -m http.server 8000 --directory docs
```

执行完成后，在浏览器中访问 `http://localhost:8000` 即可查看当前资源索引的 HTML 渲染效果。若只需要使用 Markdown 源文件，可直接在 `docs/` 目录下查看或编辑对应的索引条目。

## 贡献指南

1. 复刻本仓库到个人账户下，并在本地克隆复刻后的副本。请确保使用 Git 2.30 以上版本，并设置正确的用户信息以记录提交归属。

2. 在 `docs/` 目录下找到对应的分类索引文件，按格式规范追加或修改链接条目。每个条目必须包含 URL、标题、标签和状态字段，描述信息建议控制在 200 字以内。

3. 运行 `scripts/check_urls.py` 脚本对新增或修改的链接进行有效性检查，确保所有外链可正常访问。若遇到超时或 4xx/5xx 响应，请先核实链接是否仍然有效。

4. 提交变更时使用语义化的 Commit Message，格式为 `<type>(<scope>): <subject>`，其中 type 建议使用 `feat`、`fix`、`docs` 或 `chore`，scope 填写受影响的分类名称。提交后推送到复刻分支。

5. 在原始仓库中发起 Pull Request，并填写 PR 模板中的变更说明、验证结果与影响范围。等待至少一位维护者审核通过后，变更将被合并入主分支。

## 常见问题

**Q: 如果外部链接失效了，应该如何处理？**  
A: 当链接检查脚本报告某个地址不可访问时，首先手动在浏览器中尝试打开以排除临时网络问题。若确认链接已永久失效，请在对应的索引条目中将状态标记为 "broken"，并在描述中注明失效日期。如果找到了替代链接，可以直接替换 URL 字段，并在提交说明中记录替换原因。

**Q: 是否可以使用其他编程语言来编写辅助脚本？**  
A: 当前项目提供的脚本工具均基于 Python，但贡献者完全可以使用其他语言编写额外的工具，只要这些工具能够正确读写项目中的 Markdown 索引文件即可。建议将新工具的源码放置在 `scripts/` 目录下，并在 README 或文档中补充使用说明。

**Q: 索引文件中的标签是否可以自由定义？**  
A: 为了保持整体分类的一致性，建议优先使用项目规范中预定义的标签列表。如果现有标签无法满足需求，可以在 Pull Request 中提出新增标签的请求，并说明使用场景。维护者会评估是否将该标签纳入正式列表。在未获得批准前，可以使用 `custom-` 前缀作为临时标签。
