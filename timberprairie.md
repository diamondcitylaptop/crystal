# Olive Index

Olive Index 是一个面向开发者和技术研究人员的轻量化外链资源聚合与导航系统。本项目并非传统的网页爬虫或 RSS 阅读器，而是一个基于静态 Markdown 解析与索引构建的元数据枢纽。其核心定位在于将分散于多个技术博客、官方文档、社区讨论及代码仓库中的高价值外链进行语义化归类与状态标记，从而解决技术从业者在信息过载环境下对“可信源”与“时效性”的筛选痛点。

本项目特别适用于需要频繁查阅外部参考资料、维护私有技术知识库或构建持续集成环境中依赖链检查的团队。Olive Index 不存储实际的外部文件内容，仅维护指向这些资源的结构化索引条目，并辅以自定义的标签系统与健康度探测机制，确保用户访问的每一份参考资料均具备可追溯性与上下文说明。项目设计遵循 Unix 哲学，核心引擎以 Shell 脚本与 Python 轻量模块实现，零数据库依赖，支持纯静态部署，能够直接集成至 GitHub Actions 或 GitLab CI 等持续交付流水线中。

## 功能概览

- **多源外链归一化采集** 支持从指定 GitHub 仓库的 Markdown 文件中提取所有符合格式的 HTTP/HTTPS 超链接，自动去重并归类至预设的命名空间下。

- **语义标签自动注入** 基于链接所在文件的目录层级与上下文关键词（如 "api", "tutorial", "paper"），为每条记录生成可搜索的标签矩阵，提升检索精确度。

- **链路健康状态探测** 内置异步 HTTP 头检查模块，可配置超时与重试策略，定期标记失效或重定向链接，并以差异报告形式输出。

- **静态索引页面生成** 将解析后的结构化数据渲染为纯 HTML 目录页与 JSON API 端点，支持离线浏览与第三方工具消费。

- **增量更新与变更日志** 仅处理自上次构建以来发生变动的 Markdown 源文件，生成详细的变更记录（新增/删除/更新），便于审计与回滚。

- **自定义元数据扩展** 允许用户在源文件中通过特定注释块（如 `<!-- meta: priority=high -->`）注入额外控制指令，影响索引排序与展示权重。

- **多格式导出支持** 除 HTML 与 JSON 外，提供 CSV 与 YAML 格式的索引导出能力，便于导入至 Notion、Airtable 或自定义数据看板。

## 应用场景

- **技术团队内部知识库锚点管理** 当团队维护有大量分散的技术调研文档、第三方库评估报告或运维排障手册时，Olive Index 可充当统一的外链入口层，自动收集散落于各文档中的参考链接，并定期验证其可用性，避免知识库中堆积失效引用。

- **开源项目依赖文档的辅助审查** 开源项目维护者往往需要在 README 或贡献指南中引用大量外部资源。通过 Olive Index 对项目文档目录的持续扫描，维护者可快速获得一份所有外链的状态报表，在版本发布前主动修复断裂链接，提升项目专业度与用户体验。

- **个人技术博客的链接资产盘点** 技术博主可使用 Olive Index 定期对自己的博客仓库进行扫描，识别出所有引用的外部文章、工具站或视频教程，并生成一份“外链地图”。这既有助于读者快速找到相关扩展阅读，也方便博主在友链变更时及时更新。

- **DevOps 流水线中的质量门禁环节** 将 Olive Index 集成至 CI 流程中，作为代码合并前的检查步骤之一。当检测到文档中新增的高优先级外链（如生产环境配置文档链接）返回非 2xx 状态码时，可配置流水线失败告警，强制开发者修复或确认后再合入。

- **数据采集管道的前置校验层** 对于依赖公开数据集或第三方 API 文档的数据工程任务，Olive Index 可周期性抓取并解析数据源声明文档中的所有外链，自动提取出数据字典、接口示例或 Schema 定义文件的地址，为下游采集脚本提供动态更新的配置种子。

## 快速开始

以下步骤将指导您在本机环境中快速启动 Olive Index 核心服务，完成对示例资源库的首次索引构建。

```bash
# 克隆项目仓库至本地
git clone https://github.com/zerxonhy/olive-index.git
cd olive-index

# 安装 Python 依赖包（推荐使用虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行索引构建脚本，默认扫描 ./sources 目录下的所有 Markdown 文件
# 生成结果将输出至 ./output 目录
./bin/olive-build --source ./samples --output ./dist --check-health

# 启动本地静态预览服务（可选）
python -m http.server -d ./dist 8080
```

## 项目结构

项目源码组织遵循清晰的模块分层，各目录职责分明，便于维护与二次开发。

```
olive-index/
├── bin/                                # 可执行脚本入口
│   ├── olive-build                     # 主构建脚本，协调解析、检查与渲染流程
│   └── olive-health-check              # 独立的健康检查工具，支持单链接或批量模式
├── core/                               # 核心逻辑模块（Python 包）
│   ├── __init__.py
│   ├── extractor.py                    # Markdown 解析与链接抽取器
│   ├── classifier.py                   # 基于规则与统计的标签分类引擎
│   ├── checker.py                      # 异步 HTTP 健康状态检查器
│   └── renderer.py                     # 多格式输出渲染器（HTML/JSON/CSV）
├── lib/                                # 共享工具函数与第三方封装
│   ├── git_utils.py                    # 封装 Git 操作，用于增量获取文件变更列表
│   ├── net_utils.py                    # 网络请求重试、超时与代理配置
│   └── text_utils.py                   # 上下文关键词提取与文本清洗辅助
├── samples/                            # 示例数据源，用于快速体验与回归测试
│   ├── tech-blog.md                    # 包含多种格式外链的模拟技术博客
│   └── api-doc.md                      # 含有 JSON Schema 引用链接的示例文档
├── tests/                              # 单元测试与集成测试套件
│   ├── test_extractor.py               # 针对不同 Markdown 变体的解析测试
│   └── test_checker.py                 # 模拟网络异常的检查器健壮性测试
├── output/                             # 构建生成物的默认输出目录（gitignore）
│   ├── index.html                      # 人类可读的索引导航页
│   └── index.json                      # 机器可读的结构化索引数据
├── docs/                               # 项目完整文档（详见文档导航章节）
├── requirements.txt                    # Python 依赖锁定文件
└── README.md                           # 本文件
```

## 常见问题

**问：Olive Index 是否会存储或缓存外部链接的实际内容？**

答：不会。本项目仅维护链接的元数据（如 URL、来源文件路径、检测时间、状态码与标签）。健康检查过程仅发送 HEAD 请求以验证可达性，不会下载响应体。所有构建结果均为静态文本文件，不涉及对第三方内容的复制或托管。

**问：如何处理需要认证或带有动态参数的链接？**

答：对于需要 HTTP Basic Auth 或 Bearer Token 的链接，您可以在源 Markdown 文件中通过元数据注释块注入认证头信息（例如 `<!-- meta: auth-header="X-API-Key: xxx" -->`）。项目检查器会读取该配置并附加至探测请求。对于带有时间戳或随机参数的回调链接，建议在分类器中将其标记为 `dynamic` 类型，并降低其健康检查的告警级别。

**问：索引构建速度缓慢，如何优化？**

答：构建速度主要受网络 I/O 与 Markdown 文件数量影响。建议按以下顺序优化：1）调整 `olive-build` 的 `--parallel` 参数以增加网络检查并发数，默认值为 5；2）通过 `--exclude-pattern` 排除无需扫描的目录或文件；3）启用增量模式 `--incremental`，仅处理 Git 状态中发生变更的文件。对于大规模仓库，可考虑将源文件按子目录拆分为多个构建任务并分片运行。
