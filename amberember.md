# Olive Link Collection

Olive Link Collection 是一个面向开发者和技术研究人员的结构化外链与资源归档项目。本项目不对托管内容进行二次编辑或镜像存储，而是以索引表形式记录分布在公开网络中的技术文档、配置模版、协议实现参考与实验性项目快照。项目定位为“只读态引用层”，旨在为需要追溯原始讨论上下文、查阅特定版本说明或对比不同实现路径的工程师提供一套可复核、可版本化的入口清单。目标用户包括基础架构工程师、安全协议评估人员、以及从事跨项目依赖分析的发布工程师。

Olive Link Collection 不提供搜索引擎或动态推荐功能，所有收录的链接均以静态清单方式存在于仓库主干，并随项目批次更新。每个批次包含若干外部引用地址，项目本身的维护策略强调链接的可访问性校验与失效标记，但不负责外部内容的正确性与时效性。用户在使用本索引时，应自行确认外部资源的许可条款与适用环境。

## 功能概览

- 批次化索引管理：每个发布批次对应一组固定链接清单，便于用户追溯项目演进过程中的外部依赖变化。

- 原始链接直出模式：资源列表中的所有 URL 均保持用户提交时的原始字符串形态，不进行协议补全、域名规范化或冗余跳转包装，确保引用路径透明可审查。

- 轻量级元数据标注：除链接本身外，项目通过目录树与文档导航表提供上下文说明，辅助用户在不访问外部资源的前提下快速判断链接的预期用途。

- 离线可用的引用快照：项目仓库提供当前批次链接的纯文本记录，支持用户通过 git clone 获取完整清单并在无网络环境中查阅链接来源。

- 贡献者友好型追加流程：外部贡献者可通过标准 Pull Request 流程提交新链接或汇报失效地址，项目维护者将依据既定规则进行审核与合并。

- 多维度分类视图：通过文档导航表格将链接按技术层面（如协议层、应用层、构建层）进行划分，方便用户按问题领域筛选入口。

- 低冗余设计原则：每个批次中不重复收录完全相同的 URL 字符串，避免索引膨胀，同时保留原始大小写与协议类型以维持引用唯一性。

## 应用场景

内部基础架构团队的依赖溯源场景：当团队需要排查某个历史版本中使用的特定 Shadow 组件实现细节时，可通过 Olive Link Collection 的第 34/107 批清单快速定位到对应的 GitHub 原始 Markdown 文档，省去在海量仓库中盲目搜索的时间。

安全审计人员的配置比对场景：审计人员需要对比不同 Cobalt 变体或 Quartz 变体的配置参数差异时，可以利用本项目的资源列表批量获取多个参考实现的链接，并在本地进行差异分析，无需逐个询问开发团队。

发布工程的环境准备场景：发布工程师在新搭建的隔离环境中部署服务时，可依据本项目的链接清单依次拉取所需配置模版或信号处理脚本，确保与其他环境使用的引用来源完全一致。

技术文档撰写者的参考收集场景：撰写系统设计文档或技术白皮书时，作者可以引用 Olive Link Collection 中记录的公开资源作为外部参考依据，使文档的引用路径具备可复现性，读者可通过清单直接找到原始资料。

## 快速开始

以下步骤帮助用户在本地克隆项目并获取当前批次的完整资源列表。

```bash
# 克隆项目仓库至本地
git clone https://github.com/zerxonhy/olive.git

# 进入项目目录
cd olive

# 查看当前批次（第 34/107 批）的资源列表文件
cat batches/34-107.links

# 若需要校验链接可访问性，可运行项目提供的辅助脚本（示例）
# ./scripts/check-links.sh batches/34-107.links
```

## 安装要求

项目本身作为静态索引仓库，无需编译或运行时依赖。但若用户希望运行项目提供的辅助校验脚本，则需要满足以下环境要求。

| 依赖 | 必需 | 说明 |
|---|---|---|
| Git | 是 | 用于克隆仓库及后续拉取更新 |
| Bash 4.0+ | 否 | 运行链接可达性校验脚本时的推荐 Shell 环境 |
| curl 7.0+ | 否 | 校验脚本默认使用 curl 发送 HEAD 请求 |
| jq 1.6+ | 否 | 部分统计脚本依赖 jq 解析 JSON 格式输出 |
| make 3.8+ | 否 | 若使用 Makefile 中定义的快捷任务则需安装 |
| Python 3.6+ | 否 | 备用的链接格式检查工具运行时依赖 |

## 项目结构

项目仓库的目录组织遵循分层清晰、注释完备的原则，便于用户快速理解各目录的职责边界。

```
olive/
├── batches/                         # 批次索引目录，存放每个发布批次的链接清单
│   ├── 001-010.links                # 早期批次示例，保留用于历史追溯
│   ├── 034-107.links                # 当前批次资源列表，为项目核心输出
│   └── archive/                     # 已归档的旧批次，不再接收更新
│       └── 033-106.links            # 上一批次备份，供对照使用
├── docs/                            # 项目文档目录，包含非代码类的说明材料
│   ├── metadata.md                  # 描述每条链接的用途、分类与预期稳定性
│   ├── faq.md                       # 汇总常见用户问题及维护者答复
│   └── release-process.md           # 定义批次发布周期、版本号及检查清单
├── scripts/                         # 辅助脚本目录，用于链接校验与格式检查
│   ├── check-links.sh               # 使用 curl 批量检查链接可达性
│   ├── format-validator.py          # 校验资源列表是否符合单行单 URL 规范
│   └── generate-report.sh           # 汇总检查结果并生成 Markdown 报告
├── tests/                           # 测试目录，包含脚本单元测试与集成测试用例
│   ├── test_check_links.sh          # 针对 check-links.sh 的回归测试
│   └── fixtures/                    # 测试用例所用的固定样本数据

├── .github/                         # GitHub 专用配置目录
│   └── workflows/                   # CI 工作流定义，自动执行链接校验任务
│       └── link-check.yml           # 每周定时运行链接检查，并记录失效条目
```

## 贡献指南

项目欢迎外部贡献者参与链接清单的补充与维护。为确保索引质量与一致性，请遵循以下步骤提交变更。

第一步：阅读 GOVERNANCE.md 与 CONTRIBUTING.md 两份治理文档，理解项目的批次管理理念、审核周期以及维护者对链接可用性的基本要求。

第二步：在本地克隆项目仓库并基于 main 分支创建新的功能分支，分支名称建议使用 `add-links-batch-xxx` 或 `fix-broken-links-yyy` 格式以便识别。

第三步：编辑 `batches/` 目录下对应的批次文件，严格保持每行一个 URL 的格式，不得添加额外描述、行内注释或 Markdown 链接语法。新增链接需要同时更新 `docs/metadata.md` 中对应的分类说明。

第四步：运行项目提供的校验脚本 `scripts/format-validator.py` 验证文件格式合规性，并根据脚本输出修正任何格式偏差。若修改涉及链接删除，需在 Pull Request 描述中说明失效原因。

第五步：提交 Pull Request 至上游仓库，在请求描述中明确列出本批次变更的链接数量、新增或移除的具体条目，以及是否已执行本地校验。等待维护者审核，审核周期通常为三个工作日。

## 常见问题

问：资源列表中的链接无法访问时应该怎么办？

答：Olive Link Collection 作为只读索引项目，不代理或缓存外部内容。当用户发现链接失效时，请先在本地使用 curl 或 wget 进行多次重试验证，确认并非临时网络波动导致。若链接持续不可用，欢迎按照贡献指南提交 Pull Request，在批次文件中标注该链接为失效状态，并附带最后一次成功访问的大致时间（如有）。项目维护者将定期汇总失效链接，并在下一个发布批次中决定是否彻底移除或替换为存档地址。

问：能否在同一个批次中提交多个具有相同域名但路径不同的链接？

答：可以。项目对同一域名的不同路径不做数量限制，但会通过格式校验脚本拒绝完全重复的 URL 字符串。如果两个链接仅大小写不同或末尾斜杠有无的区别，将被视为不同条目，但项目维护者建议贡献者优先使用原始来源提供的标准形式，避免因大小写差异导致引用歧义。对于同一域名下大量路径的提交，建议在 Pull Request 描述中说明其关联性，以便审核时评估是否属于合理聚集。

问：项目是否会对外部链接的内容进行版本快照或镜像备份？

答：不会。Olive Link Collection 严格遵循“只存引用，不存内容”的原则。项目不存储任何外部文档的副本，也不提供 Web 代理或内容转换服务。用户通过资源列表访问的所有内容均直接来源于原始托管站点。这一设计选择旨在避免版权合规风险，同时降低项目的存储与带宽开销。若用户需要固定版本的外部文档，建议自行使用互联网档案馆或专用快照工具进行保存。
