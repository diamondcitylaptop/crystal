# Olive Link Collection

Olive Link Collection 是一个面向开发者与技术研究者的结构化外链与资源导航系统。本项目并非传统的网页收藏夹或书签管理器，而是一套基于 Markdown 与 Git 工作流的可维护、可审核、可版本化的资源元数据仓库。其核心定位在于将散落于 GitHub 仓库、技术博客、官方文档与社区讨论中的高价值外链，按照主题域、成熟度与用途进行二次分类与语义标注，从而降低信息检索的认知负荷，提升技术决策的效率。

项目目标用户包括但不限于：需要维护团队公共知识库的架构师、从事技术选型的研发负责人、撰写技术方案的系统设计师，以及希望系统化追踪特定技术领域生态进展的研究人员。Olive Link Collection 不试图替代搜索引擎或大语言模型，而是提供一种经过人工筛选与结构化的“资源锚点”体系，使得用户能够在特定上下文中快速定位到最相关、最权威的外部参考资料。

## 功能概览

- **分层资源索引**：资源按技术领域、应用层级与成熟度进行三级标签分类，支持快速过滤与定位。

- **锚点文件体系**：每个资源条目以独立的 Markdown 文件存储，包含资源标题、原始 URL、摘要描述、标签列表与维护状态，便于版本管理与增量更新。

- **双向引用追踪**：系统内部支持资源之间的相互引用，可追溯某一技术概念在不同外部文章中的阐述差异与演进脉络。

- **自动化元数据校验**：通过 CI 工作流自动检查每个资源条目的 URL 可访问性、摘要长度与标签格式，确保条目质量。

- **多维度视图输出**：支持将资源数据渲染为按主题聚合的概览面板、按时间排序的更新日志或按标签筛选的明细表格，适应不同使用习惯。

- **外部快照引用**：对关键外部资源，支持记录互联网存档（Internet Archive）的备份链接，增强长期可追溯性。

- **协作审核流程**：基于 Pull Request 的贡献机制，新增或修改的资源条目需经过至少一名维护者审核，确保收录标准的一致性。

## 应用场景

- **技术选型调研**：团队在评估消息队列中间件时，可通过本系统的“中间件”标签快速聚合 Apache Kafka、RabbitMQ、Pulsar 等项目的官方文档、性能对比文章与生产环境案例，大幅减少重复搜索时间。

- **新人入职培训**：新加入的研发人员可通过系统内的“入门指南”与“最佳实践”分类，系统性地了解团队常用的技术栈、开发规范与外部参考手册，加速融入团队。

- **架构评审准备**：架构师在编写系统设计文档或评审材料时，可通过系统快速检索与当前方案相关的设计模式解析、开源项目参考实现以及社区讨论中提出的潜在风险点，提升评审质量。

- **技术博客写作**：技术作者在撰写深度分析文章时，可通过系统的引用追踪功能，快速查找过往已收录的相关参考文献，确保论述的严谨性与引用来源的完整性。

- **开源项目维护**：开源项目维护者可将系统作为项目 Wiki 的外延，集中管理外部依赖的文档地址、相关生态工具链接与社区扩展列表，便于贡献者查阅。

## 快速开始

以下步骤帮助您在本地环境快速部署并运行 Olive Link Collection。

```bash
# 1. 克隆项目仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 2. 安装依赖（项目基于 Node.js 与 npm）
npm install

# 3. 启动本地开发服务器
npm run dev
```

执行上述命令后，系统默认会在本地 3000 端口启动开发服务。您可通过浏览器访问 `http://localhost:3000` 查看资源索引面板。如需构建生产环境静态文件，请执行 `npm run build`。

## 贡献指南

1. **查阅贡献手册**：在提交贡献之前，请仔细阅读 `/docs/contributing/` 目录下的详细指引，了解资源条目的元数据格式、标签规范与摘要撰写建议。

2. **复刻仓库并创建分支**：将本项目复刻（Fork）至您个人的 GitHub 账户下，然后在本地从 `main` 分支创建新的特性分支（feature branch），分支名称应简明描述本次贡献内容。

3. **新增或修改资源条目**：在 `data/entries/` 目录下新增或编辑对应的 Markdown 文件。确保所有必填字段完整，且摘要长度不超过 200 个字符。如需新增分类，请同步更新 `data/categories/` 下的索引文件。

4. **本地校验**：在提交前，请运行 `npm run validate` 命令对变更内容进行本地校验，确保格式正确且无新增死链。

5. **发起 Pull Request**：将变更推送至您的复刻仓库，然后向本仓库的 `main` 分支发起 Pull Request。在描述中清晰说明本次变更的背景、目的与影响范围。等待维护者审核，并根据反馈进行修改。

## 常见问题

**问：如果我发现一个外部资源链接已经失效，应该如何处理？**

答：您可以在本仓库的 Issues 中标记“链接失效”标签并附上具体的资源文件名，或者按照贡献指南提交 Pull Request，将该条目的状态字段更新为 `deprecated`，并在备注中说明失效情况。维护者会定期处理此类反馈，并尝试寻找可替代的更新链接。

**问：项目中的标签体系是否可以自定义扩展？**

答：可以。但为了维护整体索引的一致性，建议新增标签前先在 Discussions 中发起提议，说明新标签的定义、适用范围与预期使用频率。经过维护者确认后，新标签会被添加到标签白名单中，方可用于资源条目。个人分支不受此限制，但合并至主仓库时需遵循统一规范。
