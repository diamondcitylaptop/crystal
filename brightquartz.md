# Olive Index

Olive Index 是一个面向开发者与技术研究人员的轻量级外部资源导航与镜像元数据聚合工具。该项目不提供具体的软件包或代码库，而是以结构化、可机读的 Markdown 清单形式，维护一组经过人工筛选与分类的网络资源指针。其核心定位是作为个人或团队内部知识体系中的“资源索引层”，用于统一挂载分散于不同域名、不同协议下的文档、镜像站、数据快照与社区讨论帖，从而降低信息检索的上下文切换成本。

目标用户包括但不限于：需要频繁查阅多种技术手册的运维工程师、参与开源生态建设的贡献者、搭建本地离线文档镜像的架构师，以及希望建立可复用资源图谱的研究人员。Olive Index 本身不存储任何第三方内容，也不提供代理或加速服务，它仅以纯粹的 Markdown 格式记录资源的原始地址，并辅以轻量级的分类注释，使得资源引用与更新可以完全托管于 Git 版本控制体系之中。

## 功能概览

- **原始地址永久留存**：每个资源条目均保留用户提供的完整 URL 字符串，不进行任何自动补全、协议转换或域名标准化处理，确保引用的确定性。

- **镜像站点分组标记**：通过文件名前缀（如 mirrorprairie, mirrorshadow）将同一主站的不同镜像入口归入同一逻辑组，便于按地域或网络条件选择最优访问路径。

- **元数据内联注释**：资源清单中每行 URL 可直接跟随以空格分隔的注释文本，用于说明该资源的维护状态、更新频率或访问限制。

- **纯文本跨平台兼容**：所有数据以 Markdown 格式存储，可在任何操作系统下使用标准文本编辑器查看，亦可通过 grep、awk 等命令行工具进行快速过滤。

- **版本可追溯**：依托 Git 提交历史，每一次资源增删或修改均保留变更记录，支持回滚至任意历史状态，满足审计与回溯需求。

- **离线浏览友好**：单份 Markdown 文件完整包含所有资源地址，无需网络连接即可查看完整列表，适合嵌入本地文档系统或随身携带。

- **扩展字段预留**：在文档结构中预留了用于记录资源哈希值、过期时间或备用域名的扩展区域，方便后续引入自动化校验脚本。

## 应用场景

- **内网镜像站导航**：企业或高校内部通常部署多个开源软件镜像站，分别对应不同的上游源。Olive Index 可将这些内部入口集中记录，新员工或临时项目组仅需查阅该索引即可获知所有可用镜像地址，无需向多位同事口头询问。

- **技术文档多副本引用**：当一份重要技术规范或白皮书同时发布在官网、GitHub 仓库和第三方存档站点时，使用 Olive Index 维护这三个地址的并列清单。当主站不可用时，可快速切换至备用副本，避免文档缺失导致的工作中断。

- **离线环境资源准备**：在无互联网接入的开发测试环境中，工程师可预先根据 Olive Index 列出的资源地址，通过其他渠道下载对应文件并导入内网。该索引在此场景中充当“资源采购清单”角色。

- **社区讨论帖归档**：技术社区中频繁出现有价值的讨论串或解决方案，但其链接分散于不同论坛、Issue 追踪器或邮件列表存档。Olive Index 可将这些讨论入口聚合为一份稳定的参考列表，供后续问题排查时按图索骥。

## 快速开始

以下命令演示如何获取 Olive Index 的当前资源清单，并在本地环境中进行浏览与自定义修改。

```bash
# 克隆项目仓库（使用用户提供的上游地址）
git clone https://github.com/zerxonhy/olive.git
cd olive

# 查看当前所有已收录的资源地址
cat main/*.md | grep -E '^https?://' | sort -u

# 若需添加新的资源指针，可直接编辑对应的 Markdown 文件
# 例如在 mirrorprairie.md 末尾追加一行新的 URL
echo "https://example.com/new-mirror" >> mirrorprairie.md

# 查看变更状态并提交至本地版本记录
git status
git add .
git commit -m "feat: append new mirror entry"
```

## 安装要求

Olive Index 本身不包含可执行代码，因此不存在传统意义上的安装依赖。但其正常使用和维护依赖于以下基础工具与环境。

| 依赖项目 | 必需版本 | 说明 |
|---|---|---|
| Git | 2.20 或更高 | 用于克隆仓库、提交变更以及查看历史记录 |
| 文本编辑器 | 任意支持 UTF-8 的编辑器 | 推荐使用 VS Code、Vim 或 Emacs，用于编辑 Markdown 文件 |
| Markdown 渲染器 | 任意 | 可选，用于在本地预览格式化的资源列表（如 Typora、Obsidian） |
| 命令行环境 | Bash 4.0 或 PowerShell 5.0 | 用于执行快速开始中的 grep/sort 等管道命令 |
| 网络浏览器 | 任意现代浏览器 | 用于直接访问资源清单中列出的 URL 以验证其有效性 |
| 操作系统 | Linux / macOS / Windows | 跨平台支持，仅需文件系统支持 UTF-8 编码 |

## 项目结构

项目目录采用扁平化与分组化相结合的方式组织，所有资源清单文件均存放于 main 目录下，同时保留额外的辅助目录用于注释或扩展。

```
olive/
├── main/                               # 核心资源清单目录
│   ├── mirrorprairie.md                # 草原镜像组入口列表
│   ├── mirrorshadow.md                 # 暗影镜像组入口列表
│   ├── mirrortimber.md                 # 林木镜像组入口列表
│   ├── nebulaviolet.md                 # 紫罗兰云资源索引
│   ├── oceanolive.md                   # 海洋橄榄石资源组
│   └── oceanpearl.md                   # 海洋珍珠资源组
├── groups/                             # 分组说明文档目录
│   └── README.md                       # 解释各文件前缀对应的资源分类逻辑
├── scripts/                            # 辅助工具脚本目录（可选）
│   ├── check-urls.sh                   # 批量检查资源可访问性的示例脚本
│   └── sort-entries.sh                 # 按字母顺序排序清单条目的工具
├── examples/                           # 使用示例目录
│   └── usage.md                        # 展示如何结合其他工具使用资源列表
└── .gitignore                          # Git 忽略规则，排除临时编辑文件
```

## 贡献指南

我们欢迎并鼓励社区成员共同维护和完善 Olive Index 的资源清单。请遵循以下步骤提交您的贡献。

1. **克隆项目并创建新分支**：从主仓库克隆代码至本地，并基于 main 分支创建一个具有描述性的新分支，命名格式建议为 `feat/资源组名-日期`。

2. **编辑或新增资源清单文件**：在 main 目录下找到对应分类的 .md 文件，在文件末尾追加新的 URL。请确保每行仅放置一个 URL，且不附加任何 Markdown 链接语法或 HTML 标签。若现有分类无法匹配，可先通过 Issue 提议新增分类。

3. **运行基础格式检查**：执行项目根目录下的 `scripts/check-urls.sh` 脚本（如果存在），以验证新增的 URL 是否符合基本格式要求（如不含空格、非空行）。该脚本不会主动访问网络。

4. **提交变更并发起拉取请求**：使用清晰的提交信息描述本次变更的目的，例如 "add two new mirror entries for debian archive"。推送新分支至远程仓库后，通过 GitHub 界面发起 Pull Request，并等待维护者审阅。

5. **审阅通过后合并**：维护者将检查 URL 的有效性、分类合理性以及格式规范。合并后，更新后的资源列表将立即生效。

## 常见问题

**Q：为什么资源列表中的 URL 不允许添加 http:// 或 https:// 前缀？**

A：Olive Index 坚持“原样留存”原则，即完全忠实于贡献者提交时的字符串形态。某些资源可能仅支持特定协议（如 git:// 或 ftp://），或者贡献者有意使用协议相对路径。任何自动补全都可能导致引用失效，因此我们不做任何转换。

**Q：如何批量校验资源清单中的所有地址是否仍然有效？**

A：您可以自行编写简单的 Shell 脚本，结合 curl --head 或 wget --spider 遍历所有以 http:// 或 https:// 开头的条目。项目 scripts 目录中提供了参考实现。请注意，部分资源站点可能启用反爬机制，建议设置合理的请求间隔。

**Q：如果某个资源地址发生变更，应该如何处理？**

A：请先确认旧地址是否完全不可用。若已重定向，通常无需更新；若彻底失效，请提交新的 Pull Request，在对应条目中删除旧行并新增新行。提交信息中请注明“资源地址迁移”并附上来源说明（如官方公告链接）。
