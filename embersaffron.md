# Ocean Olive Resource Atlas

Ocean Olive Resource Atlas 是一个面向开发者与技术研究人员的结构化外链与资源聚合工具。项目不提供传统意义上的代码库或软件包，而是以精心维护的 Markdown 资源索引为核心，将分散在多个仓库、文档站点与社区论坛中的高价值技术内容进行统一归集、分类标注与版本追踪。项目定位为“技术资源的资源”，适用于需要快速定位特定领域（如分布式系统、云原生运维、性能调优、开源合规等）深度资料的场景。目标用户包括架构师、SRE 工程师、开源贡献者以及技术决策者，帮助其在信息过载的环境中高效获取可信、可追溯的参考链接与配套说明。

项目本身以静态 Markdown 文件构成，完全基于 Git 进行版本管理，每个资源条目均包含原始出处、抓取时间、内容摘要与标签体系。通过约定的文件命名与目录层级，用户可以按主题、按时间、按来源机构进行筛选与引用。Ocean Olive Resource Atlas 不替代搜索引擎，而是提供经过人工审核与社群投票的精选资源集合，确保每条外链均具备明确的技术深度或实践参考价值。

## 功能概览

- **多维度资源索引**：每个资源条目均附带主题分类、难度等级、适用技术栈与最后验证时间，支持通过 grep 或 IDE 全局搜索快速定位。

- **自动化元数据提取**：借助 CI 流水线，每次推送后自动扫描资源链接的有效性，检测 404、超时或内容变更，并在资源状态表中标记异常。

- **版本化快照引用**：对关键外链提供内容摘要的本地缓存，即使原始页面下线，用户仍可阅读预先提取的核心观点与代码片段。

- **标签与交叉引用系统**：资源间通过 `See also` 字段建立关联，形成知识图谱，便于从某篇论文跳转到相关工具实现或性能评测报告。

- **社群更新机制**：允许通过 Issue 或 Pull Request 提交新资源，经维护者审核后合并，合并过程自动触发元数据重新计算与索引重建。

- **离线阅读模式**：整个资源库可克隆至本地，无需网络即可浏览全部摘要、目录与结构化注释，适合内网环境或长时间调研场景。

## 应用场景

- **技术选型前调研**：团队计划引入新的消息队列中间件，通过 Ocean Olive Resource Atlas 的“消息系统”分类，可一次性获取 Kafka、Pulsar、RocketMQ 的官方文档、性能对比文章、生产环境踩坑记录与调优指南，大幅缩短调研周期。

- **故障排查参考**：线上出现罕见的 JVM 内存泄漏，运维人员可在资源库中检索“JVM 调优”标签，快速找到经过社群验证的 GC 日志分析工具使用教程与典型堆转储案例，辅助定位根因。

- **开源合规审查**：法务或合规团队需要审核项目依赖的第三方库许可证兼容性，资源库中的“开源协议”专题汇总了各协议全文、兼容性矩阵及 FAQ，并提供官方 SPDX 标识符对照表，减少查阅原始法律文本的时间。

- **新员工技术培训**：为新入职的开发者定制学习路径，通过资源库的“入门必读”与“进阶推荐”两类标签，组合出涵盖语言基础、框架原理、运维部署与监控告警的完整学习清单，确保知识体系覆盖全面。

- **技术博客写作素材**：技术博主撰写分布式事务相关文章时，可通过资源库快速收集 Seata、TCC、Saga 等模式的原始论文链接、开源实现地址以及已知性能瓶颈讨论，作为引用的可靠出处。

## 快速开始

以下命令将项目克隆至本地，并完成初始环境配置与索引生成。注意项目本身不依赖编译环境，仅需 Git 和标准文本处理工具。

```bash
# 克隆主仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖工具（用于元数据校验和格式检查）
# 此处以 Ubuntu/Debian 为例，其他系统请使用对应包管理器
sudo apt-get update
sudo apt-get install -y coreutils grep sed curl jq

# 运行本地索引构建脚本，该脚本会扫描所有 Markdown 文件并生成资源总表
bash scripts/build_index.sh

# 启动本地静态预览服务（Python 内置 HTTP 服务器）
python3 -m http.server 8000

# 此时在浏览器中访问 http://localhost:8000 即可查看生成的资源导航页
```

若需在本地进行资源新增或更新，建议先创建新的分支，完成修改后提交 Pull Request，详细流程请参阅贡献指南章节。

## 安装要求

项目作为静态资源库，对运行环境要求极低，但为获得完整的自动化校验与预览体验，建议满足以下依赖。若仅进行纯文本阅读，则无需安装任何额外软件。

| 依赖组件 | 必需性 | 说明 |
|---|---|---|
| Git 2.25 及以上 | 必需 | 用于克隆仓库、管理版本历史以及提交变更。 |
| Bash 4.0 及以上 | 必需 | 运行项目提供的各类辅助脚本，包括索引构建、链接检查与格式校验。 |
| GNU Coreutils | 必需 | 提供 sort、uniq、comm 等基础工具，用于资源去重与统计。 |
| curl 7.68 及以上 | 推荐 | 用于执行外链可达性检测，若缺失则跳过网络验证步骤。 |
| jq 1.5 及以上 | 推荐 | 用于解析 JSON 格式的元数据文件，若缺失则部分统计功能降级。 |
| Python 3.6 及以上 | 可选 | 仅用于启动本地 HTTP 预览服务，不影响资源数据的读取与编辑。 |

## 贡献指南

项目欢迎所有形式的贡献，包括新增资源、修正失效链接、完善分类体系以及改进文档表述。为确保协作顺畅，请遵循以下标准化流程。

1.  **创建 Issue 讨论**：在新增大量资源或调整分类结构前，请先创建一个 Issue 说明意图与具体方案，等待维护者反馈后再进行实质性修改，以避免无效劳动。

2.  **派生仓库并本地修改**：点击 GitHub 页面右上角的 Fork 按钮派生本仓库至个人账户，随后克隆派生仓库到本地，并在新功能分支（例如 `feature/add-kubernetes-networking`）上进行修改。

3.  **遵守文件格式规范**：每个资源条目必须包含 `title`（标题）、`url`（原始链接）、`category`（分类）、`tags`（标签列表）、`summary`（摘要，不少于 50 字）五个字段，并按照模板填充。新增或修改后，务必在本地运行 `scripts/validate_format.sh` 进行格式检查。

4.  **提交 Pull Request**：将本地分支推送至派生仓库，然后通过 GitHub 网页界面发起 Pull Request 到主仓库的 `main` 分支。PR 描述中请关联对应的 Issue 编号，并勾选自查清单（已运行校验脚本、已添加测试资源等）。

5.  **参与评审与迭代**：维护者或其他贡献者会在 PR 中提出修改意见，请及时响应并补充提交。所有 CI 检查（链接有效、格式规范、无合并冲突）均通过后，PR 将被合并。

## 常见问题

**问：项目中的资源链接如果失效了怎么办？**

项目维护者不会立即删除失效链接，而是会在链接旁标注 `[DEPRECATED]` 或 `[BROKEN]` 状态，并在 `metadata/link_status.json` 中记录最后检测时间。用户仍可查看该资源的摘要缓存以获得参考信息。若您是资源原所有者且希望移除链接，请通过 Issue 联系维护团队，我们会尽快处理。

**问：我可以将本项目用于商业产品中吗？**

可以。本项目采用 MIT 许可证，允许自由使用、复制、修改、合并、出版发行、散布、再授权及销售软件副本。但需注意，项目本身仅聚合外部链接，每个链接对应的原始内容仍受其各自版权或许可证约束，使用外部资源时请遵守原始网站的条款。

**问：如何确保新增资源的质量？**

所有新增资源均经过至少两位维护者的评审，评审标准包括：内容的技术深度、是否提供可复现的示例或数据、发布日期是否在 18 个月内（经典论文除外）、以及是否属于项目定义的收录范围。此外，社区投票机制允许通过 Issue 点赞或反对特定资源，当反对票超过阈值时会触发重新评估。
