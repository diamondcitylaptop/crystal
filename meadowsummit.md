# Olive Collective

Olive Collective 是一个面向开发者、技术写作团队与开源文档维护者的结构化外链与资源汇聚工具。项目本身不存储具体文档内容，而是以资源索引清单的形式，将分散在不同仓库、不同分支中的技术说明、规范文档、版本记录与操作手册进行统一梳理与归类。目标用户包括技术文档工程师、开源项目维护者、内部平台管理员以及需要频繁查阅多源技术资料的研发人员。Olive Collective 通过明确的目录约定、资源分类规则和可扩展的索引结构，帮助团队降低文档查找成本，提升跨项目协作效率，同时为后续的文档迁移、版本比对与合规审计提供清晰的入口依据。

## 功能概览

- **外链索引聚合**：支持将多个外部 Markdown 文件以链接形式收录至统一清单，并按照预设的分类规则进行分组展示，便于快速定位特定主题或模块的相关资料。

- **资源状态标记**：每条外链可附带简单的状态说明，例如稳定版本、草案阶段、已废弃或待审阅，帮助团队成员在查阅时同步了解文档的可用性与更新进度。

- **批次管理机制**：所有资源链接按照导入批次进行编号与归档，当前批次为第 22/107 批，便于追溯每次新增或变更的来源、时间和操作人员。

- **纯静态化输出**：项目核心输出为静态 Markdown 文档，无需数据库或后端服务即可完整呈现所有索引内容，降低部署与运维门槛。

- **目录树辅助导航**：提供清晰的 ASCII 目录树结构，展示资源文件在本地的组织方式，帮助新加入的维护者快速理解项目的物理布局与逻辑划分。

- **贡献流程标准化**：内置贡献指南模板，明确新增链接、更新描述或移除失效资源的具体操作步骤，减少协作过程中的沟通成本。

- **多场景适配能力**：既可作为公开开源项目的文档入口，也可用于企业内部知识库的外链管理，灵活适配不同规模与安全要求的技术环境。

## 应用场景

**技术文档团队统一入口管理**  
当文档团队同时维护多个产品线的说明文档，且各产品线的文档分别存放在不同的代码仓库或分支中时，Olive Collective 可作为统一的索引层，将所有核心文档的外链汇总至一处。团队成员只需查阅当前索引清单，即可快速跳转至所需文档，无需记忆多个仓库路径。

**开源项目外部参考归档**  
开源项目在演进过程中，常需要引用外部规范、参考实现或社区讨论记录。Olive Collective 允许维护者将这类外部资源链接集中保存，并附上批次与状态标记，确保后续版本迭代时能够快速回溯当时的参考依据，避免因外部链接失效或内容变更导致的信息丢失。

**内部平台运维手册整理**  
企业内部的技术运维团队通常拥有多套环境的管理手册、应急预案和配置说明。这些资料可能分散在 Wiki、代码仓库或共享网盘中。通过 Olive Collective 的索引机制，运维人员可以将所有相关手册的外链整合为一份结构化的资源清单，并按照环境类型或服务模块进行分类，显著提高故障排查时的资料检索效率。

**多版本文档对照查阅**  
在软件版本发布过程中，不同版本的文档往往保留在不同的分支或标签下。Olive Collective 允许在同一索引中并列列出多个版本的文档链接，并标注版本号与发布日期。技术支持人员或客户成功团队可据此快速切换查阅不同版本的说明，准确回答用户关于版本差异的咨询。

## 快速开始

以下命令演示了如何将 Olive Collective 项目克隆至本地，并完成基础安装与运行准备。请确保在执行前已满足安装要求章节所列出的依赖条件。

```bash
git clone https://github.com/zerxonhy/olive.git
cd olive
npm install
npm run build
```

执行完成后，项目将生成静态资源索引页面，默认输出至 `dist` 目录。您可以直接使用任何静态 Web 服务器托管该目录，或在本地使用开发服务器进行预览。

## 贡献指南

**第一步：查阅现有资源清单**  
在新增链接之前，请先检查资源列表章节以及 `config/categories.json` 中的分类映射，确认待添加的链接尚未被收录，或当前记录是否需要更新。

**第二步：创建或更新批次记录**  
根据当前批次编号（第 22/107 批），在 `config/batches.json` 中新增一条导入记录，注明导入日期、操作人员以及本次新增或变更的链接数量。

**第三步：编辑资源列表**  
在 `src/indexer/parser.js` 或对应的资源数据文件中，按照既定的分类规则添加新的外链条目。每个条目至少需要包含链接地址、分类标签和状态标记。

**第四步：本地构建与验证**  
运行 `npm run build` 命令执行完整构建流程，确保新添加的链接能够正确出现在生成的静态页面中。同时执行 `npm test` 运行测试套件，确认未引入任何破坏性变更。

**第五步：提交变更并发起拉取请求**  
使用 Git 提交本次变更，提交信息中需包含批次编号与简要说明。随后在 GitHub 仓库中发起拉取请求，等待项目维护者审阅与合并。

## 常见问题

**问：资源列表中的链接出现 404 错误时应该如何处理？**  
答：首先确认链接中的仓库名、分支名和文件路径是否与原始地址完全一致。由于 GitHub 上的文件可能被重命名、移动或删除，建议通过 GitHub 网页界面访问该仓库，手动验证目标文件是否存在。若文件已不存在，请在资源列表中将该条目的状态标记更新为“已失效”，并在备注中记录检查日期。若文件仍存在但路径发生变化，则使用新的正确链接替换原有条目，并在提交信息中注明本次路径修正。

**问：是否支持在资源列表中添加非 GitHub 来源的外链？**  
答：支持。Olive Collective 的核心定位是通用外链索引工具，不限制链接的来源域名。您可以在资源列表中添加任何有效的 HTTP 或 HTTPS 链接，包括但不限于 GitLab、Bitbucket、自建 Git 服务、Wiki 页面或云存储共享链接。唯一要求是链接必须为可公开访问的完整 URL，且目标内容与项目技术文档或资源整理相关。添加时请确保在分类标签中明确标注来源类型，以便团队成员区分不同平台的内容。
