# Olive Link Aggregator

Olive Link Aggregator 是一个面向技术开发者与开源项目维护者的外链资源归集与结构化导航系统。该项目不提供具体的功能代码或运行时服务，而是以 Markdown 文档仓库的形式，对分散在网络各处的技术手册、API 参考、社区讨论、配置模板和示例代码片段进行人工筛选、分类索引和语义标注。目标用户包括运维工程师、全栈开发人员、技术决策者以及刚接手遗留系统的新人。通过解决“资源发现成本高、收藏夹杂乱、关键链接遗忘”的普遍痛点，Olive 将原始 URL 转化为可维护、可审计、可共享的知识资产清单。

## 功能概览

- **按技术域分层归集**：每个资源链接依据其核心主题（如 API 网关、数据库迁移、前端构建、安全策略）被分配至对应领域目录，支持快速过滤。

- **锚点化摘要提取**：对每个收录的 URL 人工提炼 1-2 句功能摘要，并标注文件格式（Markdown、YAML、Shell 脚本等），便于预览。

- **版本与分支标记**：明确记录资源所在仓库的默认分支或标签（如 main、stable），避免追踪到过时文档。

- **交叉引用关系图**：在目录树中显式标注资源之间的依赖或补充关系（如“前置阅读”、“配套示例”）。

- **变更日志与审查状态**：每条资源记录包含最后验证日期和状态标记（有效 / 需复查 / 已失效），提升长期可维护性。

- **轻量级全文检索**：借助 GitHub 原生搜索能力，通过统一的前缀命名（如 `falcon*`、`field*`、`garden*`）实现快速定位。

- **一键导出清单**：提供脚本将当前索引导出为纯文本列表或 JSON Schema，便于集成到其他文档生成工具。

## 应用场景

**场景一：团队新人入职培训**
新成员需要快速了解公司技术栈涉及的外部依赖和内部规范。Olive 提供了一份经过整理的起点链接列表，新人可以直接从 `falconanchor.md` 开始阅读基础架构概述，避免在海量官方文档中迷失方向。

**场景二：微服务拆分前的影响分析**
在将单体应用拆分为微服务时，架构师需要重新评估现有 API 网关、服务注册中心和配置中心的使用方式。通过 `fieldbloom.md` 和 `gardenmeadow.md` 中记录的社区实践链接，团队可以快速对比不同方案的优劣，降低决策风险。

**场景三：依赖升级时的兼容性检查**
当核心框架或中间件发布大版本更新时，运维人员需要查阅大量发行说明和迁移指南。Olive 的 `falconisland.md` 和 `fieldember.md` 集中了多个版本间的破坏性变更对照表，显著缩短排查时间。

**场景四：离线环境下的文档镜像规划**
对于内网隔离环境，管理员需要提前准备完整的文档镜像。Olive 的资源列表可直接作为 wget 或 curl 批量下载的输入源，确保所有必要文档被一次性同步。

## 快速开始

以下操作基于 Linux / macOS 环境，确保本机已安装 Git 和任意文本编辑器。

```bash
# 克隆仓库
git clone https://github.com/zerxonhy/olive.git
cd olive

# 安装依赖（仅用于本地预览脚本，非运行环境）
# 本仓库无运行时依赖，若需使用清单导出工具，请安装 Python 3.9+
pip install --upgrade pip
pip install pyyaml

# 运行索引验证脚本（检查所有 Markdown 文件的链接有效性）
python scripts/validate_links.py --path ./ --output report.txt

# 启动本地预览服务（基于 Python 内置 HTTP 服务器）
python -m http.server 8000
# 浏览器访问 http://localhost:8000 查看文档索引首页
```

## 贡献指南

我们欢迎任何形式的贡献，包括但不限于新增资源链接、更新失效地址、补充摘要描述或修复文档格式。请遵循以下步骤：

1. **提交前检查**  
   在 GitHub 上 Fork 本仓库，并确保您的本地分支基于最新的 `main` 分支。运行 `scripts/validate_links.py` 确保现有链接无新增失效项。

2. **编辑或新增条目**  
   若要新增资源，请复制 `templates/resource_template.md` 到根目录，按模板填写标题、URL、摘要、分类标签和验证日期。若修改已有条目，请更新 `CHANGELOG.md` 的“未发布”章节。

3. **提交变更并创建 Pull Request**  
   提交信息请使用约定式提交格式，例如 `docs: add new resource for service mesh`。在 PR 描述中明确说明该资源解决的具体技术问题或使用场景。

4. **接受审查与合并**  
   项目维护者将检查链接有效性、摘要准确性和格式规范。如无问题，您的 PR 将被合并，同时 `CHANGELOG.md` 中的“未发布”条目将移至新版本。

5. **定期复审**  
   每季度项目维护者会批量复查所有资源链接，标记失效或过时条目。如果您发现某个链接已经不可用，请随时提交 issue 或 PR 进行更新。

## 常见问题

**问：Olive 是否可以替代浏览器书签或在线收藏服务？**  
答：Olive 不是实时同步工具，也不提供云端存储。它更适合作为一份可版本控制、可团队协作的静态资源清单。您仍然可以使用浏览器书签进行日常快速访问，但 Olive 专注于长期维护和知识沉淀，尤其适合在项目交接或架构评审时作为权威参考依据。

**问：为什么不对 URL 做自动重定向或短链处理？**  
答：本项目的核心原则是保持原始 URL 的完整性和透明度。自动重定向可能引入中间页跟踪参数或缓存失效问题，而短链会增加额外的解析延迟和信任风险。我们推荐用户直接使用原始链接，同时通过脚本定期检查响应状态码来确保可用性。

**问：如何报告资源列表中的某个链接已失效？**  
答：您可以直接在本仓库的 Issues 页面提交一个问题，标题包含失效链接的文件名（如 `fieldbloom.md`），并在正文中附上当前返回的 HTTP 状态码（如 404 或 502）。如果您有替代的有效链接，也可以一并提供。我们会尽快验证并更新相应条目。
