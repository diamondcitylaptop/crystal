# Olive Link Hub

Olive Link Hub 是一个面向开发者与技术研究人员的轻量级外链资源汇总与导航系统。该项目定位于技术社区中的“知识锚点”角色，通过结构化的 Markdown 索引机制，将分散于 GitHub 仓库中的技术笔记、配置参考、架构方案与实验记录进行集中归类与稳定挂载。项目本身不依赖数据库或后端服务，完全基于静态文件体系与 Git 版本控制，适用于个人知识库管理、团队内部技术文档导航以及开源项目中外部参考资源的快速引用。

目标用户包括需要维护大量技术链接的文档工程师、构建个人知识体系的软件工程师、以及希望以低门槛方式组织技术收藏夹的团队管理者。Olive Link Hub 解决的核心问题是：当技术参考链接数量增长后，如何通过可维护、可追溯、可协作的方式对其进行有效组织，并降低链接失效或路径变更带来的信息丢失风险。

## 功能概览

- **按主题索引的链接目录**：所有外链资源按照主题领域划分，每个 Markdown 文件对应一个独立的技术分类，便于按需查阅。

- **链接状态标记与备注**：每条链接支持附加状态标记（如有效、失效、待审）和简要说明，帮助团队成员快速判断参考价值。

- **版本化变更追踪**：依托 Git 提交历史，所有链接的新增、删除或修改均可追溯，支持回滚与变更审计。

- **轻量级全文检索**：基于 grep 或 ripgrep 可对所有链接文件进行快速关键词检索，无需启动任何额外服务。

- **自动化链接格式校验**：项目内置简单的脚本工具，可批量检查链接格式合法性，并提示可能存在的路径错误。

- **多层级目录组织**：支持二级子目录结构，允许在主题分类下进一步细分场景，适应不同规模的链接集合。

- **Markdown 原生兼容**：所有资源文件采用标准 Markdown 语法，可在 GitHub、GitLab 等主流代码托管平台直接渲染，零学习成本。

## 应用场景

- **技术文档外链附录**：当开源项目需要引用大量外部参考资料（如论文、API 文档、规范标准）时，可将所有链接集中存放于 Olive Link Hub，并在主文档中通过相对路径引用，避免主文档过于冗长。

- **团队知识库索引构建**：技术团队可将日常积累的博客文章、视频教程、工具站点等链接按主题整理为多个 Markdown 文件，并定期通过 Pull Request 进行团队评审，形成可持续维护的知识索引。

- **个人学习路线资源管理**：学习者可按照不同的学习阶段（入门、进阶、专题）建立独立的链接清单，结合备注字段记录阅读状态与个人心得，构建个人化的学习路径导航。

- **离线文档参考集散地**：对于需要在内网或无网络环境中工作的场景，可将所有链接对应的页面摘要或本地缓存路径一并记录，作为离线文档体系的外部指针层。

## 快速开始

以下命令可帮助您在本地快速启动 Olive Link Hub 实例，并开始管理您的链接资源。

```bash
# 克隆项目仓库到本地
git clone https://github.com/zerxonhy/olive.git olive-link-hub
cd olive-link-hub

# 安装项目依赖（包括链接校验工具与格式化脚本）
npm install

# 运行本地开发服务器（用于预览 Markdown 文件渲染效果）
npm run dev
```

完成上述步骤后，访问控制台输出的本地地址即可查看已收录的链接资源列表。您可以直接编辑 `main` 目录下的 Markdown 文件来增删或修改链接条目。

## 贡献指南

欢迎各类形式的贡献，包括新增链接、修正失效链接、优化文档结构以及改进脚本工具。请遵循以下步骤：

1.  **Fork 本仓库并创建新分支**：请在 GitHub 上 Fork 本项目，然后基于 `main` 分支创建一个以 `feature/` 或 `fix/` 为前缀的新分支，用于存放您的修改。

2.  **遵循 Markdown 编写规范**：所有链接条目必须使用无序列表格式，每条链接占一行，并附带简要说明。说明文字应使用中文，简洁明了，不超过 20 个字。

3.  **运行本地校验脚本**：在提交 Pull Request 之前，请务必在项目根目录运行 `npm run validate` 命令，确保所有链接格式正确，且无重复条目。若校验失败，请根据输出提示修正后再行提交。

4.  **提交 Pull Request 并描述变更**：将您的分支推送至您的 Fork 仓库，然后向本仓库的 `main` 分支提交 Pull Request。请在 Pull Request 描述中清晰说明新增或修改的链接内容及其用途，以便维护者进行评审。

5.  **等待评审与合并**：项目维护者将在 3 个工作日内对您的 Pull Request 进行评审。如无问题，您的修改将被合并至主分支；如有修改建议，维护者会在 Pull Request 中留言反馈。

## 常见问题

**问：如果某个外部链接失效了，应该如何处理？**

答：当发现链接失效时，请优先尝试在互联网上寻找该资源的最新有效地址。如果找到，请提交 Pull Request 更新链接；如果确认资源已永久移除，请在对应的 Markdown 文件中将该条目移动到 `archive/` 目录下对应的年份子目录中，并添加 `[已失效]` 标记。

**问：我能否在链接文件中添加个人备注或评分？**

答：可以。项目允许在每个链接条目后追加备注内容，格式为 `- [链接地址] 备注：您的备注文字`。备注内容建议保持客观、简洁，避免主观情绪化描述。如果备注涉及敏感信息，请将条目放在 `main/bookmarks/` 目录下，该目录默认不对外公开。

**问：项目是否支持自动检测链接可用性？**

答：项目当前未内置自动检测外部链接可用性的功能，但您可以通过 GitHub Actions 的定时任务结合 `scripts/validate-links.js` 脚本实现简单的 HTTP 状态检查。欢迎贡献者提交更完善的自动化检测方案。
